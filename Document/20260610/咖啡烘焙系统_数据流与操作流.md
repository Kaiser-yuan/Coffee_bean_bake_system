```
生豆档案 (GreenBean)         1 ──── N    采购批次 (PurchaseBatch)

烘焙批次 (RoastingBatch)     1 ──── 1    烘焙曲线 (RoastingCurve)
烘焙批次 (RoastingBatch)     1 ──── N    评价问卷 (Questionnaire)

批次复盘 (BatchReview)       1 ──── 3    烘焙提醒 (Reminder)
采购批次 (PurchaseBatch)     1 ──── N    库存调整 (InventoryAdjustment)
```

## 2. 生豆数据采集流

这是系统的**第一个入口流程**，所有后续操作的前提。

### 2.1 流程图

```
开始
  │
  ├──► 在"生豆名称"输入框中搜索
  │      │
  │      ├── 匹配到已有生豆 ──► 选择已有档案
  │      │                         │
  │      │                         ├── 自动填入：豆种、处理法、产区
  │      │                         └── 进入「新增采购批次」模式
  │      │                                │
  │      │                                └──► 填写采购信息 ──► 保存
  │      │
  │      └── 未匹配 ──► 进入「录入新豆款」模式
  │                        │
  │                        ├── 填写：豆种、处理法、产区、农场、海拔
  │                        └── 填写：采购日期、总量、含水率、供应商、批次号
  │                              │
  │                              └──► 保存（同时创建：生豆档案 + 首个采购批次）
  │
  ▼
数据变更
  ├── mockGreenBeans 新增一条记录
  ├── mockPurchaseBatches 新增一条记录
  └── 生豆管理页面列表刷新
```

### 2.2 涉及的数据对象

| 步骤 | 创建/读取对象 | 关键字段 |
|------|--------------|----------|
| 搜索 | 读取 `GreenBean[]` | name, variety, region |
| 创建新豆款 | 创建 `GreenBean` | name, variety, process, region, farm, elevation |
| 创建采购批次 | 创建 `PurchaseBatch` | greenBeanId, purchaseDate, totalWeight, moistureContent, supplier, lotNumber |

### 2.3 API 调用链

```
POST /api/green-beans          → 创建生豆档案
POST /api/purchase-batches     → 创建采购批次
```

### 2.4 后续流转

采购批次创建完成后 → 可在该批次行点击「+ 待烘计划」创建烘焙计划。

## 3. 烘焙执行流

这是系统的**核心操作流程**，覆盖从计划到完成的完整生命周期。

### 3.1 流程图

```
创建待烘计划（三个入口可选）
  │
  ├── 入口 A：生豆管理 → 采购批次行 → [+ 待烘计划]
  ├── 入口 B：烘焙分析 → [+ 新建待烘计划]
  └── 入口 C：批次复盘 → [从此创建下一次待烘计划]（携带提醒和建议）

  ▼
创建窗口
  │
  ├── 自动带入：生豆名称、采购批次、剩余库存（只读参考）
  ├── 填写：计划日期、投豆量、烘焙目标
  └── 保存 ──► mockRoastingBatches 新增一条(status=planned)

  ▼
待烘计划出现在两个位置
  │
  ├── 系统总览 → 待烘豆列表 → 按豆种分组展示
  └── 生豆管理 → 三级表格 → 第三级烘焙批次

  ▼
执行烘焙（物理操作，系统外）
  │
  └──► 烘焙完成后回到系统

  ▼
标记完成（两个入口）
  │
  ├── 入口 A：系统总览 → 待烘列表 → 勾选 [完成]
  └── 入口 B：烘焙分析 → 批次详情 → [标记完成]

  ▼
批次状态变更
  │
  ├── status: 'planned' → 'completed'
  ├── actualDate 记录完成日期
  ├── beanWeightOut 计算出豆量
  └── weightLossRate 计算失重率

  ▼
可上传 CSV（待后端对接）
  │
  ├── curveStatus: 'none' → 'parsed'
  └── 生成 RoastingCurve 数据
```

### 3.2 批次状态机

```
                    ┌──────────────┐
         创建计划    │              │    标记完成
   ───────────────►  │   planned    │ ──────────────►
                    │   (待烘)      │
                    └──────────────┘
                                              │
                                              ▼
                                       ┌──────────────┐
                                       │              │
                                       │  completed   │
                                       │  (已完成)     │
                                       └──────┬───────┘
                                              │
                          ┌───────────────────┼───────────────────┐
                          │                   │                   │
                          ▼                   ▼                   ▼
                   curveStatus          evaluationStatus     reviewStatus
                   none/parsed          none/open/closed     none/draft/done
```

### 3.3 涉及的数据对象

| 操作 | 对象 | 字段变更 |
|------|------|----------|
| 创建待烘 | `RoastingBatch` | status=planned, plannedDate, beanWeightIn, targetDescription |
| 标记完成 | `RoastingBatch` | status→completed, actualDate, beanWeightOut, weightLossRate |
| CSV 上传 | `RoastingCurve` | points[], events[] |

## 4. 曲线分析流

系统的**核心分析功能**。

### 4.1 流程图

```
进入方式
  │
  ├── 总览页 → 最近烘豆 → 点击行
  ├── 烘焙分析 → 批次表格 → 点击行
  ├── 生豆管理 → 三级行 → 点击烘焙批次
  └── 烘焙分析 → 勾选 2-4 个批次 → [对比 N 锅曲线]

  ▼
曲线分析页加载
  │
  ├── 读取 RoastingCurve.points[] → 渲染曲线
  ├── 读取 RoastingCurve.events[] → 标记关键事件
  ├── 读取 RoastingBatch → 显示 KPI 指标
  └── 读取 GreenBean → 显示生豆信息

  ▼
单锅模式交互
  │
  ├── 图例点击 → 切换指标显隐 (toggleChannel)
  ├── 悬停数据点 → 显示同一时刻全部指标值 (tooltip)
  ├── 缩放 → ECharts 内置 dataZoom（计划中）
  └── [进入复盘] → 跳转复盘页

  ▼
多锅对比模式交互
  │
  ├── 按指标显隐 → toggleChannel (全局控制某个指标在所有批次中的显隐)
  ├── 按批次显隐 → toggleBatch (控制某个批次所有曲线的显隐)
  └── 同批次同一颜色 → 颜色与批次 ID 绑定
```

## 5. 评价收集流

从批次发起问卷 → 评价者填写 → 结果汇总的完整路径。

### 5.1 流程图

```
───────────────── 烘焙师侧 ─────────────────

批次复盘页
  │
  ├── 条件：已有曲线数据
  ├── 点击 [发起公开评价问卷]
  │
  ▼
Questionnaire 创建
  │
  ├── status = 'open'
  ├── shareCode 生成唯一码
  ├── expiresAt 设置截止日期
  └── 批量次的 evaluationStatus → 'open'

  ▼
分享链接
  │
  ├── 评价管理页 → 复制链接
  ├── 评价详情页 → 复制链接 + 二维码
  └── URL: /eval/:shareCode

───────────────── 评价者侧 ─────────────────

访问公开链接 (手机/PC)
  │
  ├── 问卷页面加载
  ├── 不显示曲线、豆商描述、他人评价
  │
  ▼
填写评价
  │
  ├── 7 个维度五星评分（全部必填）
  ├── 风味描述多选
  ├── 自由备注（可选）
  ├── 称呼 + 身份（可选）
  │
  ▼
提交
  │
  ├── 验证：所有维度必须有分
  ├── POST → 创建 CuppingEvaluation
  ├── Questionnaire.submissionCount +1
  └── 显示感谢页

───────────────── 烘焙师侧 ─────────────────

查看结果
  │
  ├── 评价管理 → 问卷列表 → 点击行
  ├── 评价详情页
  │   ├── 各维度平均分 + 进度条
  │   ├── 高频风味标签云
  │   └── 评价明细表
  │
  └── 批次复盘页 → 评价反馈摘要（自动生成）

关闭问卷
  │
  ├── 入口 A：评价管理页 → [关闭]
  ├── 入口 B：批次复盘页 → [关闭问卷]
  └── Questionnaire.status → 'closed'
```

### 5.2 评分维度

| 维度 | 类型 | 取值 | 含义 |
|------|------|------|------|
| dryFragrance | 质量 | 1-5 | 干香 |
| wetAroma | 质量 | 1-5 | 湿香 |
| acidity | 强度 | 1-5 | 酸感（越高越酸） |
| sweetness | 质量 | 1-5 | 甜感 |
| bitterness | 强度 | 1-5 | 苦感（越高越苦） |
| aftertaste | 质量 | 1-5 | 回味 |
| overallPreference | 偏好 | 1-5 | 综合喜好 |

## 6. 复盘迭代流

把曲线数据 + 评价反馈转化为下一锅改进方案。

### 6.1 流程图

```
进入复盘 (从曲线分析页 [进入复盘])
  │
  ▼
加载复盘数据
  │
  ├── 读取 BatchReview（可能为空）
  ├── 读取 RoastingBatch → 批次概览
  ├── 读取 Questionnaire → 评价摘要
  └── 读取 CuppingEvaluation[] → 汇总统计

  ▼
步骤 1：填写个人初始复盘
  │
  ├── 文本区域，自由编写
  ├── 时机：评价到达前先写
  ├── 保存 → personalReviewAt 记录时间戳
  └── 个人复盘不被综合复盘覆盖

  ▼
步骤 2：查看评价反馈摘要
  │
  ├── 有评价 → 系统生成摘要文本
  ├── 无评价 → 提示发起问卷 / 等待反馈
  └── 评价到达后可更新综合复盘

  ▼
步骤 3：编写综合复盘
  │
  ├── 结合曲线数据 + 评价反馈
  ├── 保存 → comprehensiveReviewAt 记录时间戳
  └── 后续可更新

  ▼
步骤 4：记录下一锅调整建议
  │
  └── 保存

  ▼
步骤 5：设置烘焙提醒（最多 3 条）
  │
  ├── #1 优先级最高
  ├── #2 次之
  ├── #3 最低
  ├── 每条：简短 + 明确 + 可执行
  └── 保存

  ▼
步骤 6：创建下一次待烘计划
  │
  └── [从此创建下一次待烘计划]
      ├── 自动带入：当前生豆、采购批次
      ├── 自动带入：投豆量
      ├── 携带：下一锅建议 + 提醒内容
      └── 跳转至烘焙分析页
```

### 6.2 复盘时序

```
时间线：

 烘焙完成 (actualDate)
    │
    ├──► 当日/次日：填写个人初始复盘 (personalReview)
    │
    ├──► 之后几天：发起评价问卷 → 收集反馈
    │
    ├──► 评价截止后：查看评价摘要
    │
    └──► 随时：编写综合复盘 + 设置提醒
         │
         └──► 下一锅：带走提醒和建议
```

## 7. 系统总览数据流

### 7.1 数据聚合路径

```
DashboardYearData
  │
  ├── totalRoasts          ← 统计年内 status=completed 的批次数量
  ├── totalBeanVarieties   ← 统计年内涉及的生豆 ID 去重数
  ├── totalInputWeight     ← 累加年内已完成批次的 beanWeightIn
  ├── avgWeightLossRate    ← 后端计算年内已完成批次的平均失重率
  │
  ├── monthlyRoasts[]      ← 按月份分组统计
  ├── varietyDistribution[]← 按豆种分组统计

  ├── pendingBatches[]     ← 筛选 status=planned 的批次（限 5 条）

API: GET /api/dashboard?year=2026
```

### 7.2 年份切换时的刷新链

```
用户点击 ‹ › 切换年份
  │
  ├── store.currentYear 更新

  └── renderCharts() 重新绑定 ECharts 数据
```

## 8. 库存计算流

### 8.1 计算公式

```
剩余库存 = 采购总量 - 已完成烘焙消耗 + 库存调整量

其中：
  - 采购总量 = PurchaseBatch.totalWeight
  - 已完成烘焙消耗 = Σ(b.beanWeightIn) where b.status='completed' AND b.purchaseBatchId = pb.id
  - 库存调整量 = Σ(adj.amount) where adj.purchaseBatchId = pb.id
```

### 8.2 库存变化触发时机

| 操作 | 库存变化 | 说明 |
|------|----------|------|
| 新增采购批次 | totalWeight 增加剩余库存 | 增量 |
| 标记烘焙完成 | beanWeightIn 扣减库存 | 消耗 |
| 库存调整 | amount 增/减库存 | 盘点修正 |

> 库存由后端计算，前端只读展示。前端不直接做库存计算。

### 8.3 库存指示规则

| 剩余量 | 颜色 | 含义 |
|--------|------|------|
| > 5000g | 绿色 | 库存充足 |
| 1 - 5000g | 黄色 | 库存偏低，建议补货 |
| ≤ 0 | 红色 | 库存耗尽 |

## 9. 用户操作路径汇总

### 9.1 主工作流（高频路径）

```
每日工作路径：

生豆管理
  └── 展开采购批次 → [创建待烘计划]
       │
       ▼
烘焙分析
  └── 筛选找到今天的计划 → 勾选 → 执行烘焙（物理）
       │
       ▼
系统总览 → 待烘列表 → 勾选 [完成]
       │
       ▼
烘焙分析 → 点击刚完成的批次 → 曲线分析页 → 查看曲线
       │
       ▼
曲线分析页 → [进入复盘] → 填写个人初始复盘
       │
       ▼
批次复盘页 → [发起公开评价问卷] → 粘贴链接分享
       │
       ▼
（几小时后或次日）
       │
       ▼
评价管理 → 查看评价详情 → 查看评分和风味分布
       │
       ▼
批次复盘页 → 查看评价摘要 → 编写综合复盘 → 设置提醒
       │
       ▼
[从此创建下一次待烘计划] → 带走提醒 → 循环
```

### 9.2 所有可能的操作路径

| 从哪里 | 操作 | 到哪里 |
|--------|------|--------|
| 导航栏 / | 点击 📊 | 系统总览 |
| 系统总览 → 最近烘豆 | 点击行 | 曲线分析（单锅） |
