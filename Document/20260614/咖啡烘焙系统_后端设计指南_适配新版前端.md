# 咖啡烘焙系统后端设计指南（适配新版前端）

> 编写日期：2026-06-12  
> 技术架构：Python 3.12 + FastAPI + Pydantic 2 + SQLAlchemy 2 + Alembic + PostgreSQL 17  
> 目标：为修改后的 Vue 前端提供真实、稳定、可追溯的业务能力  
> 优先级：本文作为后续后端实现的最新业务基线

---

## 1. 后端目标

后端不是简单的 SQL 转换器。它需要负责：

- 保存不可丢失的业务事实。
- 执行库存、状态、问卷和提醒复制等业务规则。
- 将复杂关系聚合成前端可以直接展示的数据。
- 解析并保留 Kaleido 原始 CSV。
- 保证重复请求、并发操作和异常情况下的数据正确性。
- 支撑完整迭代闭环。

```text
生豆档案
-> 采购批次
-> 待烘计划
-> 完成烘焙与库存扣减
-> CSV 解析与曲线分析
-> 问卷与评价
-> 复盘与下一锅提醒
-> 新待烘计划
```

## 2. 架构原则

### 2.1 第一版采用模块化单体

```text
Vue
-> Nginx
-> FastAPI Router
-> Service
-> Repository
-> SQLAlchemy
-> PostgreSQL
```

第一版不引入：

- 微服务。
- Redis。
- Celery。
- Kafka。
- Kubernetes。
- 向量数据库和 RAG。

2 核 4G 服务器足以运行当前架构。

### 2.2 前后端职责边界

后端负责：

- 权限与登录。
- 数据校验。
- 状态机。
- 库存计算与并发控制。
- 派生指标计算。
- CSV 文件存储和解析。
- 聚合查询。
- 分享链接生成。
- 数据持久化。

前端负责：

- 表单交互。
- 页面状态和加载状态。
- 图表渲染。
- 用户操作确认。
- 使用后端返回的状态和可执行操作。

前端不得自行决定：

- 剩余库存。
- 失重率。
- 发展率。
- 资料完整度。
- 养豆天数。
- 问卷是否过期。
- 当前用户是否允许执行某个状态操作。

## 6. 状态机与业务规则

### 6.1 烘焙批次状态机

```text
planned -> completed
planned -> voided
completed -> planned
completed -> voided
```

规则：

- `planned -> completed`：要求实际烘焙时间和实际投豆量，校验库存。
- `completed -> planned`：取消完成，库存消耗自动恢复。
- `completed -> voided`：不再计入库存消耗和统计。
- `voided` 默认不可直接恢复，第一版可暂不提供恢复接口。

### 6.2 批次完成与资料完整度分离

完成批次不要求：

- 出豆量。
- CSV。
- 问卷。
- 评价。
- 复盘。

详情和列表接口动态返回：

```json
{
  "completeness": {
    "missing_output_weight": true,
    "missing_curve": true,
    "missing_evaluation": true,
    "missing_review": true,
    "is_complete": false
  }
}
```

### 6.3 库存规则

```text
remaining_weight_grams
= total_weight_grams
- SUM(completed and valid roasting batch input_weight_grams)
+ SUM(inventory adjustment amount_grams)
```

所有库存读取必须使用同一聚合函数或数据库查询，禁止不同接口分别实现不同公式。

完成批次时：

```text
开启事务
-> SELECT purchase_batch FOR UPDATE
-> 重新计算库存
-> 校验实际投豆量
-> 更新批次状态
-> 提交
```

### 6.4 出豆量与失重率

```text
weight_loss_percent
= (input_weight_grams - output_weight_grams)
/ input_weight_grams
* 100
```

规则：

- 出豆量由用户实际称重后录入。
- 出豆量为空时，失重率为空。
- 出豆量不能大于投豆量。
- 出豆量只能为正数。

### 6.5 问卷规则

- 只有 `completed` 批次可以创建问卷。
- 不要求 CSV 或复盘存在。
- 一个批次可以拥有多个历史问卷。
- 同一批次第一版最多允许一个 `open` 问卷。
- 已关闭或已过期问卷禁止提交。
- 分享码必须不可预测。
- 分享 URL 由后端生成并返回。

### 6.6 评价规则

- 综合喜好必填，值为整数 `1..5`。
- 其他评分允许 `NULL`。
- 酸感和苦感是强度，不是好坏。
- 平均分忽略 `NULL`。
- 每个维度单独返回有效样本量。
- 提交评价时计算养豆天数。

### 6.7 复盘规则

- 一个批次最多一份复盘。
- 个人初始复盘和综合复盘独立保存。
- 来源提醒最多三条，优先级 `1..3` 唯一。
- 从复盘创建下一锅时，在同一事务中复制选中的提醒快照。

## 10. Kaleido CSV 解析设计

### 10.1 解析器识别

首行必须识别：

```text
KLDO data file V101
```

分段：

```text
[{PARAMETERS}]
[{EVENT}]
[{DATA}]
```

禁止直接将整个文件交给普通 `csv.DictReader`。

### 10.2 时间规则

```text
elapsed_seconds = Time / 1000
```

不能减去第一条数据的 `Time`。

### 10.3 解析流程

```text
临时保存文件
-> 校验哈希和大小
-> 创建 curve_file 记录
-> 标记 parsing
-> 识别 KLDO V101
-> 解析参数、事件和数据点
-> 计算阶段指标与控制变化
-> 原子替换当前生效 roasting_curve
-> 标记 parsed
```

失败时：

- 保留原始文件。
- `parse_status = failed`。
- 保存错误码和用户可读错误信息。
- 不覆盖旧的有效曲线。

### 10.4 首个固定验收样本

使用 `260530_9.csv`：

```text
数据点数：257
最后时间：385.280 秒
回温点：59 秒 / 111°C
转黄点：207 秒 / 166°C
一爆开始：322 秒 / 194°C
出豆：385 秒 / 202°C
发展时间：63 秒
发展率：约 16.36%
```

---

### 10.5 解析结果分层

后端必须区分三类数据，避免后续将算法提示误认为设备事实。

#### 原始事实

直接来自 CSV，不修改：

- `PARAMETERS` 参数。
- `EVENT` 事件时间和事件温度。
- `DATA` 中全部采样点。
- 原始文件哈希、大小和存储路径。

#### 确定性派生指标

依据固定公式计算：

- 脱水、梅纳、发展阶段时长。
- 各阶段占比。
- 各阶段平均 RoR。
- 最大 RoR。
- 火力、风门和滚筒的变化节点。
- AUC，并保存计算基准和算法版本。

第一版 AUC 规则：

```text
对 BT 高于 100°C 的部分进行梯形积分
单位：°C·min
```

#### 数据质量提示与分析提示

属于规则判断，不修改原始事实：

- 事件温度与临近采样点差异过大。
- 关键事件缺失。
- 时间不递增。
- 采样点异常跳变。
- 一爆后 RoR Crash / Flick 初步提示。

提示记录至少包含：

```text
code
severity
message
related_event
algorithm_version
```

Crash、Flick 和风味影响属于待校准分析能力。第一版只能返回提示，不得返回确定性风味结论。

### 10.6 解析模块实现建议

推荐代码职责：

```text
KaleidoV101Parser
- 识别文件和分段
- 解析参数、事件和数据点
- 输出标准化原始事实

CurveMetricCalculator
- 计算阶段指标、阶段平均 RoR、AUC 和控制变化

CurveQualityChecker
- 生成数据质量和曲线分析提示

CurveComparisonService
- 校验可对比曲线
- 生成对齐时间
- 计算基准批次与对比批次的指标差异
```

解析器不得直接依赖数据库模型。建议先生成 Pydantic 领域对象，通过 Service 在事务中写入数据库，便于使用 CSV 文件做纯单元测试。

单文件解析继续在 FastAPI 进程内同步执行即可。2 核 4G 第一版不需要 Celery；接口应限制文件大小和单次对比数量。

建议限制：

```text
单 CSV 最大：20MB
单次对比批次：2 至 5 个
默认返回采样点：原始精度
未来数据量变大后：增加降采样参数
```

### 10.7 多锅对比接口响应规范

推荐接口：

```text
GET /curve-comparisons?batch_ids={id1},{id2}&align_by=charge
```

请求规则：

- `batch_ids` 至少 2 个，第一版最多 5 个。
- 第一个 `batch_id` 为基准批次。
- `align_by` 支持 `charge`、`yellowing`、`first_crack_start` 和 `drop`。
- 所有参与对比的曲线必须解析成功。
- 使用指定事件对齐时，任一曲线缺少该事件则返回明确错误。

每个曲线点同时返回：

```json
{
  "sample_index": 1,
  "elapsed_seconds": 128.28,
  "aligned_seconds": -78.72,
  "bean_temp_celsius": 136,
  "environment_temp_celsius": 197,
  "ror_celsius_per_minute": 11.06,
  "heating_power_percent": 80,
  "smoke_damper_percent": 40,
  "roller_percent": 60
}
```

含义：

- `elapsed_seconds` 是该锅真实经过时间，永远不修改。
- `aligned_seconds` 是相对于对齐事件的时间。
- 按入豆对齐时，`aligned_seconds` 等于 `elapsed_seconds`。

响应建议结构：

```json
{
  "base_batch_id": "uuid-1",
  "align_by": "charge",
  "series": [
    {
      "batch": {},
      "events": [],
      "metrics": {},

    }
  ],
  "metric_differences": [],
```

## 11. 文件存储与备份

建议目录：

```text
/data/uploads/roast-curves/{year}/{month}/{roasting_batch_id}/{curve_file_id}.csv
```

规则：

- 文件名使用系统 ID，不使用用户原始名称作为存储名。
- 原始文件永久保留。
- 校验扩展名、大小、哈希和格式。
- 文件下载必须经过权限校验。
- 数据库和文件目录每天备份。
- 第一版可以使用服务器磁盘，OSS 作为后续备份目标。

## 12. 幂等、事务与并发

必须支持幂等的操作：

- 标记完成。
- 取消完成。
- 关闭问卷。
- CSV 重复上传。
- 评价提交的网络重试。

建议写请求支持：

```text
Idempotency-Key
```

必须使用事务：

- 新建豆款与首购。
- 完成、取消完成和作废批次。
- 创建问卷。
- 替换复盘提醒。
- 从复盘创建下一锅并复制提醒。

## 13. 安全设计

- 后台接口需要认证。
- 公开问卷接口无需登录，但使用不可预测分享码。
- 公开评价接口增加请求频率限制。
- 上传文件限制大小，第一版建议 `20MB`。
- 不信任文件名、MIME 和 CSV 内容。
- 日志中不记录密码、Token 和完整公开分享码。
- CORS 只允许配置的前端域名。
- `.env` 不提交 Git。

## 14. 测试重点

### 14.1 库存

- 待烘计划不扣库存。
- 完成批次扣库存。
- 重复完成不重复扣库存。
- 取消完成恢复库存。
- 作废完成批次恢复库存。
- 修改实际投豆量后库存正确。
- 并发完成不能超额扣库存。

### 14.2 曲线

- `260530_9.csv` 解析结果完全符合验收样本。
- 重复文件能够识别。
- 无效 KLDO 文件返回明确错误。
- 解析失败不覆盖当前有效曲线。
- 缺失事件保持为空。

### 14.3 问卷

- 未完成批次不能创建问卷。
- 没有 CSV 的已完成批次可以创建问卷。
- 已关闭或过期问卷不能提交。
- 未评价字段保存为 `NULL`。
- 养豆天数计算正确。

### 14.4 复盘

- 初始复盘与综合复盘互不覆盖。
- 最多保存三条提醒。
- 创建下一锅正确复制提醒快照。
