---

## 1. 修改目标

当前前端已经完成主要页面、基础对象关系和大部分操作入口，能够作为系统原型继续开发。

本轮修改的主要目标不是增加更多页面，而是确保：

1. 前端展示与真实业务规则一致。
2. Mock 数据结构可以平滑替换为后端 API。
3. 生豆、采购、库存、烘焙、曲线、评价、复盘能够形成完整闭环。
4. 曲线页面符合狙击手 M1 / Kaleido CSV 的真实数据格式。
5. GitHub Pages 仅作为静态演示，不让用户误以为数据已经持久化。

系统最终闭环：

```text
生豆档案
-> 采购批次
-> 创建待烘计划
-> 完成烘焙并扣减库存
-> 补录出豆量
-> 上传并解析 Kaleido CSV
-> 曲线分析与对比
-> 个人初始复盘
-> 发起评价问卷
-> 收集评价并汇总
-> 综合复盘
-> 下一锅建议与提醒
-> 创建下一次待烘计划
```

---

## 2. 当前实现中值得保留的部分

以下设计已经符合规划，后续修改时应保留：

- 生豆档案 `1 -> N` 采购批次。
- 采购批次 `1 -> N` 烘焙批次。
- 烘焙批次只保存 `purchaseBatchId`，不重复保存 `greenBeanId`。
- 生豆录入时，同名候选项展示品牌、产季、处理法和产区。
- 生豆档案已经包含品牌、产季和豆商风味描述。
- 采购批次已经包含采购价格。
- 出豆量允许后续补录。
- 投豆量和出豆量都存在时才计算失重率。
- 评价管理页面只负责汇总和管理，不创建问卷。
- 评价问卷支持“未评价 / 不确定”。
- 各维度平均分会忽略未评价数据，并展示有效人数。
- 评价记录包含冲煮方式、饮用温度和饮用形式。
- 曲线页面默认同时显示温度、RoR、火力、风门和滚筒。
- 系统总览的待烘列表已经按照生豆档案 / 豆款分组。

## 4. P0：必须在接口设计前修正

### 4.1 明确 GitHub Pages 只是静态演示环境

#### 当前问题

所有数据都保存在前端 Mock 数组中：

```text
mockGreenBeans
mockPurchaseBatches
mockRoastingBatches
mockQuestionnaires
mockEvaluations
```

刷新页面后，新增和修改的数据会丢失。其他人打开评价链接时，也无法读取发起者浏览器内创建的问卷。

#### 修改建议

在 GitHub Pages 演示版本顶部增加轻量提示：

```text
演示环境：当前数据为模拟数据，刷新页面后不会保存。
```

所有涉及持久化的成功提示，在演示版中改成：

```text
已在当前演示会话中保存
```

不要在 GitHub Pages 上宣称已经支持真实问卷分发、真实库存管理或数据备份。

#### 验收标准

- 用户能够明确知道当前为演示环境。
- 不会误以为问卷链接能够发送给其他人使用。
- 不会误以为刷新页面后数据仍然存在。

### 4.2 修正仓库中的旧业务文档

#### 当前问题

仓库内旧版数据流和功能文档仍然存在以下错误描述：

- 标记完成后自动计算出豆量。
- 问卷必须在已有曲线后发起。
- 所有评分维度必须填写。
- 待烘计划按照豆种分组。
- 烘焙批次与曲线固定 `1 -> 1`。
- 使用“转换点”，而不是“转黄点”。

这些旧文档可能再次影响数据库与接口设计。

#### 修改建议

直接修改原始文档，使其与最新修改指南保持一致，而不是只保留一份独立修改说明。

关系基数统一为：

```text
生豆档案 1 -> N 采购批次
采购批次 1 -> N 烘焙批次

烘焙批次 1 -> 0..N 原始曲线文件
烘焙批次 1 -> 0..1 当前生效曲线
烘焙批次 1 -> 0..N 评价问卷
烘焙批次 1 -> 0..1 批次复盘
批次复盘 1 -> 0..3 烘焙提醒
```

#### 验收标准

- 仓库内所有文档使用相同对象名称和业务规则。
- 文档中不再出现自动计算出豆量。
- 文档中不再将曲线或复盘作为创建问卷的前置条件。
- 所有关键事件统一使用“转黄点”。

### 4.3 将业务状态与资料完整度分开

#### 当前问题

`BatchDataCompleteness` 已经定义，但页面没有真正使用。

烘焙列表当前只用 `C / E / R` 圆点表示曲线、评价和复盘状态，没有清晰展示：

- 当前批次是待烘、已完成还是已作废。
- 已完成批次还缺少哪些资料。

#### 修改建议

烘焙列表增加两列：

```text
业务状态：
- 待烘
- 已完成
- 已作废

资料完整度：
- 缺出豆量
- 缺 CSV
- 缺评价
- 缺复盘
- 资料完整
```

`C / E / R` 可以保留为辅助信息，但不应代替文字状态。

资料完整度由前端根据接口返回字段展示，不作为批次业务状态。

#### 验收标准

- 用户可以快速区分待烘、已完成和已作废批次。
- 已完成但缺 CSV 的批次仍然显示为“已完成”。
- 用户能够快速找到需要补录的数据。

## 5. P1：补齐核心操作闭环

### 5.1 实现库存重新计算与库存流水展示

#### 当前问题

`remainingStock` 当前是随机生成或创建采购批次时直接赋值的静态展示值。

标记烘焙完成后，库存不会扣减；取消完成、作废或修改投豆量时，也无法恢复库存。

#### 正确规则

```text
采购批次剩余库存
= 采购总量
- 有效且已完成烘焙批次的实际投豆量合计
+ 库存调整量合计
```

#### 前端修改

采购批次详情增加“库存流水”入口：

| 时间 | 类型 | 关联对象 | 变化量 | 结果 |
|---|---|---|---:|---:|
| 2026-05-01 | 采购入库 | PB-001 | +5000g | 5000g |
| 2026-05-03 | 烘焙消耗 | RB-001 | -500g | 4500g |
| 2026-05-04 | 盘点调整 | 调整记录 | -20g | 4480g |

前端只展示后端计算结果，不直接修改 `remainingStock`。

必须覆盖：

- 创建待烘计划：不扣库存。
- 标记完成：扣减实际投豆量。
- 取消完成：恢复库存。
- 作废已完成批次：恢复库存。
- 修改实际投豆量：重新计算库存。
- 更换采购批次：原批次恢复，新批次扣减。
- 新增库存调整：重新计算。

#### 验收标准

- 所有库存变化都能解释来源。
- 前端没有直接编辑剩余库存的输入框。
- 创建计划不会提前扣库存。

---

### 5.2 补齐批次状态操作

#### 当前问题

虽然类型中定义了 `voided`，但前端没有完整的状态操作和状态逆转流程。

#### 修改建议

烘焙批次详情或行操作菜单增加：

```text
待烘批次：
- 标记完成
- 编辑计划
- 作废计划

已完成批次：
- 补录出豆量
- 上传 / 替换 CSV
- 取消完成状态
- 作废批次

已作废批次：
- 查看历史信息
- 恢复为待烘，按权限决定是否支持
```

危险操作使用二次确认，并说明对库存的影响。

#### 验收标准

- 状态变化路径明确。
- 状态逆转会触发库存重新计算。
- 已作废数据不会进入烘焙统计和库存消耗。

### 5.3 完成“复盘 -> 下一锅计划”提醒携带

#### 当前问题

创建下一次待烘计划时，只携带了：

- 当前采购批次。
- 当前投豆量。
- 下一锅建议。

没有携带最多三条烘焙提醒，也没有记录提醒来源。

#### 修改建议

点击“创建下一次待烘计划”时，先打开确认弹窗：

```text
采购批次
计划日期
计划投豆量
下一锅目标
需要携带的提醒，默认全部勾选
```

创建成功后，将提醒复制为下一锅的提醒快照：

```text
sourceReviewId
sourceRoastingBatchId
targetRoastingBatchId
priority
content
```

下一锅详情页面顶部显示：

```text
来自上一锅 RB-001 的提醒
#1 一爆后降火速度放慢
#2 发展率目标 18%
#3 出豆温度尝试提高 2°C
```

#### 验收标准

- 创建下一锅时可以查看和选择提醒。
- 下一锅保存的是提醒快照，不会随上一锅复盘修改而变化。
- 能够追溯提醒来源。

### 5.4 修正问卷创建条件和生命周期

#### 当前问题

当前曲线页面和 Mock API 没有严格检查批次是否已完成，待烘批次理论上也能创建问卷。

关闭问卷后，没有同步更新烘焙批次的评价状态；问卷过期时间也没有真正校验。

#### 修改建议

创建问卷按钮显示规则：

```text
待烘批次：
- 不显示创建问卷按钮

已完成批次且未创建问卷：
- 显示“发起评价问卷”

存在进行中问卷：
- 显示“评价进行中”
- 提供复制链接和查看结果

问卷已关闭：
- 显示“评价已关闭”
- 允许创建新问卷
```

问卷生命周期：

```text
open -> closed
open -> expired
```

评价管理仅提供：

- 查看结果。
- 复制链接。
- 关闭问卷。
- 查看关联烘焙批次。

#### 验收标准

- 只有已完成批次能够创建问卷。
- 问卷状态与批次评价状态保持一致。
- 过期和关闭问卷无法继续提交评价。
- 一个批次可以创建多个历史问卷。

## 6. P2：落地真实 Kaleido 曲线分析

### 6.1 更新曲线字段模型

根据真实 `260530_9.csv`，曲线点建议使用：

```ts
interface CurvePoint {
  sampleIndex: number
  elapsedMs: number
  elapsedSeconds: number
  beanTempCelsius?: number
  environmentTempCelsius?: number
  rorCelsiusPerMinute?: number
  targetTempCelsius?: number
  heatingPowerMode?: string
  heatingPowerPercent?: number
  smokeDamperPercent?: number
  rollerPercent?: number
  powerStatus?: string
}
```

第一版前端展示字段：

```text
BT -> 豆温
ET -> 环境温度
RoR -> 升温率
HP -> 火力
SM -> 风门
RL -> 滚筒
```

`SV`、`HPM` 和 `PS` 可以先保留在详情或 Tooltip 中。

---

### 6.2 更新关键事件模型

统一事件类型：

```text
charge               入豆
turning_point        回温点
yellowing            转黄点
first_crack_start    一爆开始
first_crack_end      一爆结束
second_crack_start   二爆开始
second_crack_end     二爆结束
drop                 出豆
```

不再使用：

```text
turning = 转换点
```

事件允许缺失，不允许用 `0` 代替未记录事件。

#### 验收标准

- 页面统一显示“转黄点”。
- 一爆结束和二爆等未记录事件不会显示为时间零。
- 事件 Tooltip 同时显示时间和温度。

### 6.3 按 V3 方案改造单锅曲线页面

单锅曲线推荐布局：

```text
顶部：
- 批次、生豆、日期
- 总时长、预热温度、出豆温度
- 转黄时间、一爆时间、发展时间、发展率

主图区：
- 豆温 BT：左轴
- 环境温度 ET：左轴
- RoR：右轴，绿色虚线
- 三个阶段使用浅色背景

控制区：
- 火力 HP
- 风门 SM
- 滚筒 RL
- 标记控制变化时间和前后值
```

阶段背景：

```text
入豆 -> 转黄：脱水阶段
转黄 -> 一爆开始：梅纳阶段
一爆开始 -> 出豆：发展阶段
```

RoR 规则：

- 与 BT、ET 放在同一主图。
- 使用右侧独立坐标轴。
- 设备未输出有效 RoR 的前段显示为空，不使用零值曲线。
- 图例明确显示“RoR · 右轴”。

#### 验收标准

- 用户可以在同一时间点观察温度和 RoR。
- 用户可以直接看到火力变化后的曲线响应。
- 三个阶段时长和占比能够直接读取。

### 6.5 改进多锅曲线对比

多锅对比建议提供两种模式：

```text
两锅精细对比：
- 同一指标固定使用同一种颜色
- 第一个批次使用实线
- 第二个批次使用虚线
- 适合观察相同指标的曲线形状差异

三锅及以上概览：
- 每锅一种主色
- 默认只显示 BT 和 RoR
- 用户按需打开 ET、火力、风门和滚筒
- 避免同时展示过多难以识别的线条
```

增加对齐方式：

```text
按入豆时间对齐
按转黄点对齐
按一爆开始对齐
按出豆时间对齐
```

对比页增加差异摘要：

- 阶段时长差。
- 发展率差。
- 出豆温度差。
- 阶段平均 RoR 差。
- AUC 差。
- 火力操作节点差。
- 关键事件时间差。

#### 验收标准

- 用户能够比较两锅或多锅同一阶段的差异。
- 曲线数量较多时仍然能够识别不同批次。
- 图例能够明确解释颜色代表指标、实线和虚线代表锅次。

### 6.6 双 CSV 对比页面落地规范

本节依据 `260530_9.csv` 与 `260530_10.csv` 的真实解析和对比结果制定。

#### 页面默认状态

首次进入对比页时：

```text
默认对齐：按入豆时间对齐
默认显示：BT、ET、RoR、火力、风门、滚筒
第一个批次：实线
第二个批次：虚线
BT：红色
ET：蓝色
RoR：绿色，使用右轴
火力：洋红色
风门：黄色
滚筒：青色
```

不允许仅依靠颜色区分两个批次。批次还必须通过实线、虚线和图例文字识别。

#### 页面布局

```text
顶部工具栏
- 当前参与对比的批次
- 按入豆、转黄、一爆开始、出豆对齐
- BT、ET、RoR、操作量显隐开关

阶段区
- 每锅独立显示脱水、梅纳、发展阶段条
- 显示各阶段时长

主图区
- BT、ET 和 RoR 同图
- 显示转黄、一爆开始和出豆事件
- 鼠标悬浮时同时显示两锅同一对齐位置的数据

控制区
- 火力、风门、滚筒阶梯线
- 操作量使用与主图相同的时间轴

右侧摘要区
- 两锅核心指标
- 第二锅减第一锅的差异
- 关键事件时间差
- 初步异常提示
```

#### 对齐规则与文案

- 按入豆对齐时，横轴表示真实烘焙经过时间。
- 按转黄、一爆开始或出豆对齐时，横轴表示相对事件时间。
- 对齐点左侧显示负时间，右侧显示正时间。
- 页面必须提示：“事件对齐用于比较阶段内曲线形状，不代表真实操作时间相同。”
- Tooltip 同时展示“对齐时间”和每锅“真实经过时间”。

前端不自行修改曲线点的原始 `elapsedSeconds`。对齐后的时间使用后端返回的 `alignedSeconds`，或者仅在图表适配层临时计算。

#### 差异摘要展示规则

差异统一定义为：

```text
差异 = 对比批次 - 基准批次
```

右侧摘要至少显示：

- 总时长。
- 脱水、梅纳、发展阶段时长。
- 发展率。
- 阶段平均 RoR。
- AUC，必须显示计算基准，例如 `BT > 100°C`。
- 回温、转黄、一爆开始和出豆的时间差。

差异不能简单用红色表示增加、绿色表示减少，因为增加或减少不一定代表好坏。默认使用中性色；只有系统存在明确规则或人工判断后，才显示“建议关注”。

#### 数据质量提示

当事件标记值和曲线采样值存在明显差异时，前端只提示，不自动修正。

例如：

```text
第 10 锅入豆事件温度：146°C
第 10 锅首个 BT 采样值：154°C
```

页面提示：

```text
入豆事件温度与首个豆温采样值存在差异，请核对事件标记或探针记录。
```

所有算法判断都应标注性质：

```text
事实：CSV 原始值和直接计算结果
提示：基于规则发现的数据质量问题
建议：需要结合杯测和人工判断
```

## 7. P3：数据治理与体验增强

### 7.1 让标准词表真正驱动录入表单

#### 当前问题

系统设置存在标准词表页面，但生豆录入仍然使用代码中的固定常量：

```text
BEAN_PROCESSES
VARIETY_OPTIONS
REGION_OPTIONS
```

停用词条不会影响录入表单，也没有新增词条入口。

#### 修改建议

录入页面的选择项应通过接口读取标准词表：

```text
GET /api/terms?category=process&active=true
GET /api/terms?category=variety&active=true
GET /api/terms?category=region&active=true
```

设置页面增加：

- 新增词条。
- 编辑显示名称。
- 停用 / 启用。
- 查看使用次数。

历史数据已经使用的词条可以停用，但不能删除。

#### 验收标准

- 设置页停用词条后，新录入表单不再显示。
- 历史记录仍然正常展示停用词条。
- 特殊处理法可以选择“其他”并填写说明。

---

### 7.2 修正系统总览聚合数据

#### 当前问题

月度烘焙频率目前使用随机数，与年度总锅数可能不一致。

#### 修改建议

所有总览指标必须来源于同一批已完成烘焙数据：

```text
已完成锅数
已烘豆款数
总投豆重量
平均失重率
月度烘焙频率
豆种分布
产区分布
```

平均失重率只使用投豆量和出豆量均已录入的批次。

#### 验收标准

- 月度锅数之和等于年度已完成锅数。
- 图表与列表使用相同筛选口径。
- 切换年份后所有指标同步变化。

### 7.3 加强新建待烘计划校验

创建待烘计划时需要校验：

- 必须选择采购批次。
- 投豆量必须大于零。
- 投豆量不得超过采购批次当前可用库存。
- 计划日期不能为空。
- 切换生豆后自动重置采购批次选择。

建议在生豆管理中的采购批次行创建计划，作为最高频入口；烘焙分析页的“新建待烘计划”保留为辅助入口。

#### 验收标准

- 不会创建没有采购批次的待烘计划。
- 不会因切换生豆而继续使用旧采购批次。
- 超过库存时明确提示并禁止提交。

---

### 7.4 新建生豆和首个采购批次需要原子提交

#### 当前问题

当前前端分别调用：

```text
创建生豆档案
创建采购批次
```

如果第二步失败，会产生没有采购批次的孤立生豆档案。

#### 修改建议

后端提供组合接口：

```text
POST /api/green-beans/with-first-purchase
```

前端一次提交生豆档案和首个采购批次，由后端事务保证同时成功或同时失败。

已有生豆再次采购仍使用：

```text
POST /api/purchase-batches
```

#### 验收标准

- 新建豆款时不会产生半完成数据。
- 保存失败后页面能够保留用户填写内容。
