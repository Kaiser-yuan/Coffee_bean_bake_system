# 咖啡烘焙系统前后端联调指南

> 编写日期：2026-06-12  
> 前端：Vue 3 + TypeScript + Pinia + ECharts  
> 后端：FastAPI + PostgreSQL  
> 目标：逐步将当前 Mock 前端替换为真实后端，并完成核心闭环验收

---

## 1. 联调原则

联调不是等后端全部完成后一次性替换 Mock，而是按业务闭环逐模块推进。

推荐顺序：

```text
统一接口规范
-> 标准词表
-> 生豆与采购
-> 烘焙批次与库存
-> CSV 与曲线
-> 问卷与评价
-> 复盘与下一锅
-> 系统总览
```

每个模块完成后必须：

1. 前端停止直接读取该模块的 Mock 数组。
2. 所有写操作改为调用 API。
3. 刷新页面验证数据仍然存在。
4. 验证错误状态和空状态。
5. 完成该模块的端到端验收。

## 3. 前端 API 层改造

### 3.1 禁止页面直接引用 Mock 数据

当前页面存在：

```ts
import {
  mockGreenBeans,
  mockPurchaseBatches,
  mockRoastingBatches
} from '../mock'
```

联调时应替换为：

```text
src/api/http.ts
src/api/auth.ts
src/api/terms.ts
src/api/greenBeans.ts
src/api/purchaseBatches.ts
src/api/roastingBatches.ts
src/api/curves.ts
src/api/questionnaires.ts
src/api/reviews.ts
src/api/dashboard.ts
```

页面只调用 API 模块或 Pinia Store。

### 3.2 HTTP 客户端统一行为

统一处理：

- API 基础地址。
- Token 或 Cookie。
- JSON 请求头。
- `request_id`。
- 401 登录失效。
- 网络错误。
- 业务错误。
- 文件上传进度。

建议错误对象：

```ts
interface ApiError {
  code: string
  message: string
  details?: Record<string, unknown>
  requestId?: string
}
```

## 4. 请求与响应约定

### 4.1 字段命名

后端接口统一使用 `snake_case`。

前端选择一种方式并保持一致：

1. TypeScript 类型也使用 `snake_case`；或
2. 在 API 层统一转换为 `camelCase`。

禁止部分页面使用 `beanWeightIn`，另一部分直接使用 `input_weight_grams`。

推荐第一版前端保持 `camelCase`，在 API 层集中转换。

### 4.2 时间

后端返回带时区 ISO 8601：

```text
2026-06-12T19:30:00+08:00
```

前端：

- 展示时转换为用户本地时区。
- 提交时发送带时区时间。
- 不使用只有日期的字符串代替实际烘焙时间。

### 4.3 数值与空值

- 未录入值返回 `null`，不返回 `0`。
- 重量统一为克。
- 金额统一为分。
- 温度统一为摄氏度。
- 时长统一为秒。
- 比例返回百分数值，如 `16.36`。

### 4.4 列表分页

请求：

```text
?page=1&page_size=20
```

响应：

```json
{
  "items": [],
  "page": 1,
  "page_size": 20,
  "total": 0
}
```

### 4.5 错误响应

```json
{
  "code": "INSUFFICIENT_INVENTORY",
  "message": "采购批次剩余库存不足",
  "details": {
    "available_grams": 400,
    "required_grams": 500
  },
  "request_id": "uuid"
}
```

前端根据 `code` 决定交互，根据 `message` 展示用户提示。

## 5. 联调前的契约冻结

每个接口开始联调前，双方确认：

- URL 和 HTTP 方法。
- 请求字段、类型、必填规则。
- 响应字段和空值规则。
- 错误码。
- 权限。
- 幂等规则。
- 状态变化。
- 对库存或其他对象的影响。

建议 FastAPI 自动生成 OpenAPI，并将：

```text
/docs
/openapi.json
```

作为接口契约来源。

前端可根据 OpenAPI 后续生成类型，但第一版可以手工维护。

GET   /api/v1/admin/terms

- 已存在的日晒豆款仍正常展示。
- 已使用词条无法物理删除。

## 7. 第二阶段联调：生豆与采购

### 接口

```text
GET  /api/v1/green-beans/matches?name=
POST /api/v1/green-beans/with-first-purchase
GET  /api/v1/green-beans/tree
POST /api/v1/green-beans/{id}/purchase-batches
GET  /api/v1/purchase-batches/{id}
```

### 豆款匹配联调

输入名称后，后端返回：

```json
[
  {
    "id": "uuid",
    "name": "耶加雪菲 G1",
    "brand": "豆商 A",
    "harvest_season": "2025",
    "processing_method": "水洗",
    "region": "耶加雪菲"
  }
]
```

前端必须让用户明确选择：

```text
使用已有豆款并新增采购
创建新豆款和首个采购批次
```

### 验收

- 创建新豆款和首购后刷新页面，数据仍存在。
- 第二步失败时不会留下孤立豆款。
- 同名不同产季豆款不会自动合并。
- 为已有豆款新增采购时不重复提交豆款基础字段。

## 8. 第三阶段联调：烘焙批次与库存

### 接口

```text
POST  /api/v1/roasting-batches
GET   /api/v1/roasting-batches
GET   /api/v1/roasting-batches/{id}
POST  /api/v1/roasting-batches/{id}/complete
POST  /api/v1/roasting-batches/{id}/reopen
POST  /api/v1/roasting-batches/{id}/void
PATCH /api/v1/roasting-batches/{id}/output-weight
GET   /api/v1/purchase-batches/{id}/inventory-ledger
POST  /api/v1/purchase-batches/{id}/inventory-adjustments
```

### 创建待烘计划

前端提交：

```json
{
  "purchase_batch_id": "uuid",
  "planned_at": "2026-06-15T10:00:00+08:00",
  "planned_input_weight_grams": 500,
  "target_description": "突出花香"
}
```

后端校验：

- 采购批次存在。
- 计划投豆量大于零。
- 当前可用库存足够。
- 创建计划不扣库存。

### 标记完成

前端提交：

```json
{
  "roasted_at": "2026-06-15T10:05:00+08:00",
  "input_weight_grams": 500
}
```

成功后前端必须刷新：

- 当前批次。
- 待烘列表。
- 采购批次库存。
- 系统总览摘要。

### 验收矩阵

| 操作 | 批次状态 | 库存结果 |
|---|---|---|
| 创建计划 | planned | 不变化 |
| 完成批次 | completed | 扣减实际投豆量 |
| 重复完成 | completed | 不重复扣减 |
| 取消完成 | planned | 恢复库存 |
| 作废完成批次 | voided | 恢复库存 |
| 补录出豆量 | completed | 库存不变化，计算失重率 |

## 9. 第四阶段联调：Kaleido CSV 与曲线

### 接口

```text
POST /api/v1/roasting-batches/{id}/curve-files
GET  /api/v1/curve-files/{id}
POST /api/v1/curve-files/{id}/reparse
GET  /api/v1/curve-files/{id}/download
GET  /api/v1/roasting-batches/{id}/curve
GET  /api/v1/curve-comparisons
```

### 文件上传流程

```text
前端选择文件
-> multipart/form-data 上传
-> 后端返回 curve_file 和 parse_status
-> 前端轮询解析状态
-> parsed 后请求曲线
-> failed 时展示错误和重新解析入口
```

状态轮询建议：

```text
每 1 秒查询一次
最长等待 30 秒
完成或失败后停止
```

### 单锅曲线响应要求

响应需要直接支持 V3 曲线页面：

```json
{
  "summary": {
    "total_time_seconds": 385.28,
    "preheat_temp_celsius": 206,
    "drop_temp_celsius": 202,
    "development_time_seconds": 63,
    "development_ratio_percent": 16.36
  },
  "stages": [],
  "events": [],
  "points": [],
  "control_changes": [],
  "source_file": {}
}
```

### 固定验收样本

上传 `260530_9.csv` 后验证：

- 数据点 `257`。
- 最后时间 `385.280` 秒。
- 转黄点 `207` 秒。
- 一爆开始 `322` 秒。
- 发展时间 `63` 秒。
- 发展率约 `16.36%`。
- 未记录事件为 `null` 或不返回，不使用零。
- RoR 前段无效零值由前端显示为空。

## 10. 第五阶段联调：问卷与评价

### 接口

```text
POST /api/v1/roasting-batches/{id}/questionnaires
GET  /api/v1/questionnaires
GET  /api/v1/questionnaires/{id}
POST /api/v1/questionnaires/{id}/close
GET  /api/v1/public/questionnaires/{share_code}
POST /api/v1/public/questionnaires/{share_code}/evaluations
```

### 关键规则

- 待烘批次创建问卷返回 `ROASTING_BATCH_NOT_COMPLETED`。
- 已完成且无 CSV 的批次可以创建问卷。
- 分享 URL 使用后端返回值，前端禁止手工拼接。
- 公开问卷接口不要求登录。
- 关闭或过期问卷提交评价时返回明确错误。
- 其他评分未评价时提交 `null`，不提交 `0`。

### 评价提交

```json
{
  "evaluator_name": null,
  "evaluator_type_term_id": "uuid",
  "brewing_method_term_id": "uuid",
  "drink_temperature": "hot",
  "drink_style": "black",
  "dry_fragrance_score": null,
  "wet_aroma_score": 4,
  "acidity_intensity_score": 3,
  "sweetness_score": 4,
  "bitterness_intensity_score": 2,
  "aftertaste_score": 4,
  "overall_preference_score": 4,
  "flavor_term_ids": ["uuid"],
  "notes": "回味干净"
}
```

### 跨设备验收

1. 后台创建问卷。
2. 手机或另一个浏览器打开 `share_url`。
3. 提交评价。
4. 后台评价详情出现新记录。
5. 每个维度有效人数和平均分正确。
6. 养豆天数正确。

## 11. 第六阶段联调：复盘与下一锅

### 接口

```text
GET   /api/v1/roasting-batches/{id}/review-overview
PATCH /api/v1/roasting-batches/{id}/review/personal
PATCH /api/v1/roasting-batches/{id}/review/comprehensive
PATCH /api/v1/roasting-batches/{id}/review/suggestion
PUT   /api/v1/roasting-batches/{id}/review/reminders
POST  /api/v1/roasting-batches/{id}/next-roast-plan
```

### 前端保存策略

- 每个区域单独保存，避免覆盖其他复盘内容。
- 保存成功后使用后端返回的最新复盘对象更新页面。
- 提醒使用整体替换接口，一次保存最多三条。

### 创建下一锅验收

1. 填写下一锅建议。
2. 保存三条提醒。
3. 点击创建下一次计划。
4. 弹窗选择计划日期、投豆量和需要携带的提醒。
5. 创建成功后跳转新批次详情。
6. 新批次展示提醒快照和来源批次。
7. 修改旧复盘提醒，不影响新批次快照。

## 13. 前端加载、空状态和错误状态

每个 API 页面必须覆盖：

```text
初始加载
加载成功且有数据
加载成功但为空
业务错误
网络错误
无权限
```

写操作按钮规则：

- 请求期间禁用，防止重复提交。
- 成功后显示明确反馈。
- 失败时保留用户填写内容。
- 库存不足、问卷过期等业务错误使用后端 message。
- 未知错误展示 `request_id` 便于排查。

## 14. 缓存与刷新策略

第一版不需要复杂前端缓存。

写操作完成后按影响范围刷新：

| 写操作 | 需要刷新 |
|---|---|
| 新增采购 | 生豆树、采购详情 |
| 创建待烘计划 | 批次列表、总览待烘列表 |
| 完成 / 取消 / 作废 | 批次、库存、总览 |
| 补录出豆量 | 批次、总览平均失重率 |
| 上传 CSV | 文件状态、批次资料完整度、曲线 |
| 创建 / 关闭问卷 | 批次、评价管理、复盘总览 |
| 提交评价 | 问卷详情、评价统计、复盘摘要 |
| 创建下一锅 | 批次列表、总览待烘列表 |

## 15. 联调错误码建议

| 错误码 | 前端处理 |
|---|---|
| `VALIDATION_ERROR` | 标记具体字段 |

| `CURVE_FILE_DUPLICATED` | 提示已上传相同文件 |

| `QUESTIONNAIRE_EXPIRED` | 公开页显示问卷已过期 |
| `IDEMPOTENCY_CONFLICT` | 提示操作已经处理 |

## 18. 推荐联调里程碑

- 完成、取消完成和作废可用。

- `260530_9.csv` 上传解析成功。
- V3 曲线页面展示正确。

- 跨设备问卷可填写。
- 评价统计与养豆天数正确。

- 下一锅建议与提醒可以复制。
- 新计划展示提醒来源。
- 系统总览聚合一致。
