## 3. P1-1：重写 Auto 时间推断

### 问题定位

- `backend/app/services/bulk_import.py:124-136`：文件名只解析日期并固定为本地 00:00。
- `backend/app/services/bulk_import.py:197-201`：Auto 在命中文件名后立即返回，不再使用首锅时间和顺序间隔。
- `backend/app/services/bulk_import.py:308`：顺序来自上传顺序，没有解析 `_1`、`_2` 锅次后缀。

当前最小复现结果：

```text
260621_1.csv -> 2026-06-20 16:00:00+00:00, source=filename
260621_2.csv -> 2026-06-20 16:00:00+00:00, source=filename
SAME_TIME=True
```

### 修改要求

1. 将“日期推断”和“时间推断”拆开，不再让文件名策略直接返回完整 datetime。
2. 日期优先级：CSV 日期、文件名日期、手工日期、lastModified 本地日期。
3. 时间优先级：CSV 时间、用户首锅时间 + 锅次间隔、lastModified 本地时间。
4. 新增文件名锅次解析，例如 `260621_2.csv -> order=2`。
5. 当文件存在锅次后缀时，使用锅次排序；没有后缀才回退上传顺序。
6. 45 分钟间隔移入 Settings，例如 `ROAST_SPACING_MINUTES=45`，不要保留模块常量。
7. 预览中保留可编辑 datetime-local；commit 必须使用用户修改后的 ISO 8601 值。
8. 时间来源建议拆为 `roasted_date_source` 和 `roasted_time_source`；若暂不改表，至少允许 `filename+upload_order` 这样的组合来源。

### 必补测试

- Auto + 无 CookDate + 文件名 `_1/_2`：时间相差 45 分钟。
- 上传顺序 `_2/_1`：仍按锅次得到 `_1 < _2`。
- CookDate 有完整时间：不得被首锅时间覆盖。
- File.lastModified：保持绝对时间，不重复按上海时区解释。
- datetime-local 修改后提交并重新读取：页面时间不偏移。

## 4. P1-2：关闭匿名标准词写入并实现限流

### 问题定位

- `backend/app/api/v1/evaluations.py:49-60`：公开接口调用 `get_or_create_value`，任意 brew method 和 flavor 都会写入标准词表。
- `backend/app/api/v1/evaluations.py`：`Request`、`settings` 已导入，但没有任何 429 判断。
- `backend/app/repositories/terms.py:44`：导入了 `pg_insert` 却未使用。
- `backend/app/repositories/terms.py:53-58`：冲突后 `rollback()` 会回滚当前请求的整个事务，不是安全的并发 upsert。
- `backend/tests/integration/test_bulk_import_integration.py:553-564`：所谓并发测试只是同一 session 串行调用两次。

### 修改要求

1. 公开评价只允许 active 标准词：
   - 推荐请求传 `brew_method_term_id`、`flavor_term_ids`。
   - 兼容旧前端时，可按 `category + value + is_active=true` 查询。
   - 不存在或 inactive 时返回 422，绝不创建新词。
2. 新增独立 `free_flavor_description` 文本字段；自由描述不进入 `standard_terms`。
3. `flavor_notes` 的每个元素增加 `max_length=128`，不要静默截断。
4. `evaluator_type`、`drink_temperature`、`drink_form` 改为受控 Literal/Enum，而不只是长度限制。
5. 管理端仍需要 get-or-create 时，真正使用 PostgreSQL：

```python
stmt = (
    pg_insert(StandardTerm)
    .values(category=category, value=value, is_active=True)
    .on_conflict_do_nothing(index_elements=["category", "value"])
    .returning(StandardTerm.id)
)
```

冲突后只重查，不允许对共享 session 调用全事务 `rollback()`。

6. 将 `PUBLIC_EVALUATION_RATE_LIMIT_PER_MINUTE` 接入公开 POST。
7. 单进程部署可先采用内存限流；若以后增加多 worker，改用 Redis 或 PostgreSQL 原子计数。
8. 重复提交可使用 `questionnaire_id + 哈希后的客户端标识`，只存哈希，不保存原始 IP。

### 必补测试

- 未知 flavor/brew method 返回 422，标准词数量不变。
- inactive term 返回 422。
- 自由风味描述可保存但不创建 term。
- 单字段和列表元素超长均返回 422。
- 1 分钟超过阈值返回 429。
- 使用两个独立 AsyncSession 并发创建同一管理端词条，只保留一条且不返回 500。

## 7. P1-5：修复 Vite base 与问卷分享链接

### 问题定位

- `frontend/vite.config.ts:7` 使用 `process.env.VITE_BASE_PATH`；Vite 不会把 `.env.local` 自动注入 config 的 `process.env`。
- 当前生产构建仍输出 `/assets/...` 根路径。
- 当前 `backend/.env` 设置为 `PUBLIC_FRONTEND_BASE_URL=http://localhost:5173/#/eval`。
- `generate_share_url()` 又拼接 `/#/eval/{code}`，形成重复路径。
- 现有单测按当前配置自我拼接，无法发现重复路由。

### 修改要求

1. Vite config 使用 `loadEnv`：

```ts
import { defineConfig, loadEnv } from 'vite'

export default defineConfig(({ mode }) => {
  const env = loadEnv(mode, process.cwd(), '')
  return {
    plugins: [vue()],
    base: env.VITE_BASE_PATH || '/',
  }
})
```

2. `PUBLIC_FRONTEND_BASE_URL` 只包含站点 base，不包含 hash 路由：
   - 本地：`http://localhost:5173`
   - GitHub Pages：`https://kaiser-yuan.github.io/Coffee_bean_bake_system`
3. 分享 URL 只在一个地方生成。前端优先使用后端 `share_url`，不要另写第二套规则。
4. 增加精确断言测试：根路径、GitHub Pages 子路径、末尾斜杠、禁止出现两个 `#/eval`。
5. 构建后检查 `dist/index.html` 的资源前缀。
