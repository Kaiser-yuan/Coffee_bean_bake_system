## 4. 修改一：统一烘焙时间与时区

### 当前问题

CSV CookDate、文件名日期和手工输入都是本地时间，但后端直接标记为 UTC。上海时间 09:30 可能显示为 17:30。

### 后端修改

- 在 Settings 增加 app_timezone，默认 Asia/Shanghai。
- 使用 zoneinfo.ZoneInfo 解析无时区时间。
- CSV、文件名、manual 和 upload_order 得到的 datetime 先附加本地时区，再转换 UTC。
- File.lastModified 是 epoch milliseconds，本身是绝对时间，直接按 UTC 解析。
- 数据库存储统一使用 timestamptz/UTC。

建议配置：

    APP_TIMEZONE=Asia/Shanghai

### 前端修改

- datetime-local 展示时使用浏览器本地时间。
- 提交时转换为 ISO 8601。
- 页面显示时间来源：CSV、文件名、文件时间、手工或上传顺序。

### 验收

- [ ] CookDate 2026-06-21 09:30 在页面显示 09:30。
- [ ] 提交后重新进入页面仍是 09:30。
- [ ] manual 09:30 往返不偏移。
- [ ] lastModified 往返不偏移。
- [ ] 数据库存储值带时区。

## 5. 修改二：同日多锅自动分时

### 当前问题

文件名 260621_1.csv 和 260621_2.csv 只能解析出同一天。如果 CSV 没有 CookDate，两个批次可能得到相同时间。

### 修改要求

- 分开推断日期和时间。
- 日期优先级：CSV 日期、文件名日期、手工日期、lastModified 日期。
- 时间优先级：CSV 时间、lastModified 时间、首锅时间加顺序间隔。
- 解析文件名后缀作为锅次顺序。
- 用户提供首锅时间时，按顺序自动生成后续时间。
- 默认间隔可配置，第一阶段可继续使用 45 分钟。
- 预览表允许逐行修正。

### 验收

- [ ] 删除 fixture CookDate 后，两文件仍得到不同时间。
- [ ] 后缀 1 的时间早于后缀 2。
- [ ] 上传顺序打乱时，可按文件锅次重新排序。
- [ ] 用户修改时间后 commit 使用修改值。

## 6. 修改三：历史采购归档库存模型

### 当前问题

历史补录必须选择采购批次。使用普通采购流程补建已耗尽旧采购，会把原采购重量计入当前库存，而 archive-only 烘焙不扣库存，造成虚假库存。

### 数据模型

建议给 purchase_batches 增加：

- inventory_tracking_mode：normal 或 historical_archive。
- opening_stock_grams：系统启用时的期初可用库存。

规则：

- normal 默认 opening_stock_grams 等于 total_weight_grams。
- historical_archive 可以保留 total_weight_grams，但 opening_stock_grams 默认 0。
- 库存事实计算从 opening_stock_grams 起算。
- 旧数据库迁移时，现有采购 opening_stock_grams 回填 total_weight_grams。

建议新增 Alembic 迁移 0004_purchase_batch_inventory_opening。

### 后端接口

- 新建采购批次请求增加 inventory_tracking_mode 和 opening_stock_grams。
- 历史补录页面可以创建 historical_archive 采购批次。
- purchase_in 流水的 change_grams 使用 opening_stock_grams，不盲目使用总采购重量。

### 前端修改

- 历史补录页支持选择已有采购批次或新建历史归档采购。
- 明确展示原采购重量、期初库存和当前剩余库存。
- 历史归档采购显示标签。

### 验收

- [ ] 录入原采购 5000g、期初库存 0g 后，当前库存为 0。
- [ ] 历史曲线可以挂到该采购批次。
- [ ] historical_backfill 默认不改变库存。
- [ ] normal 采购仍按采购重量入库。

## 7. 修改四：公开评价数据治理

### 当前问题

匿名评价可以把冲煮方式和风味词直接写入标准词表，可能造成词表污染、超长字段错误和并发唯一约束错误。

### 修改要求

- 标准词选择只接受 active term ID 或受控 value。
- 自由风味描述保存到独立文本字段，不自动进入 standard_terms。
- flavor、brew_method、evaluator_type、drink_temperature 和 drink_form 设置长度限制。
- get-or-create 使用 PostgreSQL ON CONFLICT 或捕获唯一约束后重查。
- 增加公开评价频率限制。
- 可选增加 questionnaire + 浏览器标识维度的重复提交限制。

### 验收

- [ ] 匿名提交不能创建任意标准词。
- [ ] 自由风味描述可以保存。
- [ ] 超长输入返回 422。
- [ ] 并发相同词不返回 500。
- [ ] 高频提交返回 429。
- [ ] 正常匿名评价仍不要求登录。

## 10. 修改七：批量任务生命周期和资源限制

### 取消及过期

新增：

    POST /api/v1/bulk-import-jobs/{job_id}/cancel

规则：

- 只有 previewed 可以取消。
- committed 不能取消。
- previewed 超过 24 小时自动取消或删除。
- 前端关闭已预览弹窗、重新预览时调用 cancel。

### 上传限制

新增配置：

    BULK_UPLOAD_MAX_FILES=20
    BULK_UPLOAD_MAX_TOTAL_BYTES=104857600

要求：

- 使用 settings，不在接口硬编码。
- 超过单文件、文件数量或总大小立即返回 422。
- 拒绝时不创建 job。

### schema 校验

- actual_input_weight_grams 必须大于 0。
- output_weight_grams 必须大于 0 且不超过投豆量。
- 服务层明确区分 None 和 0。
