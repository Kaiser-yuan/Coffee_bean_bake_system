## 2. 本次修复的关键问题

### 2.1 多 CSV commit 时间类型错误

原实现使用：

```python
item.model_dump(mode="json")
```

这会把 Pydantic 已解析的 `datetime` 转回字符串，最终导致 asyncpg 报错：

```text
expected a datetime.date or datetime.datetime instance, got 'str'
```

现已改为：

```python
item.model_dump(mode="python")
```

### 2.2 曲线 calculation_version 字段过短

数据库原字段为：

```text
roasting_curves.calculation_version varchar(16)
```

实际写入值为：

```text
curve-analysis-v1
```

该值超过 16 个字符，会导致所有真实曲线入库失败。现已新增迁移：

```text
backend/alembic/versions/20260621_0003_widen_curve_calculation_version.py
```

字段已扩为 `varchar(32)`。

### 2.3 批量导入事务和重复提交

当前实现已增加：

- 导入 job 行锁，防止并发重复 commit。
- URL 采购批次与 job 采购批次一致性检查。
- 普通批量入口与历史补录入口的 mode 一致性检查。
- 同一次上传中的重复哈希检查。
- commit 前再次检查数据库中是否已有相同文件。
- 单 item 使用 savepoint，失败时不留下半条批次、曲线或库存流水。
- 出豆量必须大于零且不能超过投豆量。
- 迁移后清理失败文件。

### 2.4 浏览器文件修改时间

前端预览请求现在会为每个文件发送：

```text
client_last_modified=<File.lastModified>
```

后端按文件顺序解析 epoch milliseconds，使 `file_last_modified` 时间推断策略真正可用。

### 2.5 前端真实 API

真实模式已补齐以下客户端和 service：

```text
dashboard
curves
questionnaires
evaluations
reviews
green bean context
standard terms
roasting batches
```

公开问卷页面不再请求需要登录的生豆树，而是直接使用公开问卷接口返回的豆名和烘焙日期。

## 6. 多 CSV 页面联调

入口：

```text
生豆管理
→ 展开生豆
→ 展开采购批次
→ 批量 CSV 生成
```

操作步骤：

1. 选择至少两个 Kaleido CSV。
2. 设置默认投豆量，例如 550g。
3. 保持“影响库存”开启。
4. 点击解析预览。
5. 检查每行解析状态、时间来源、总时长、发展率和 warnings。
6. 重复文件应显示“重复”，不能提交。
7. 库存不足时提交按钮应禁用。
8. 点击提交。
9. 生豆树中应出现相同数量的 completed 烘焙记录。
10. 每条记录都应可以打开曲线。

验收公式：

```text
提交后库存
= 提交前库存
- 所有 inventory_effective=true 行的实际投豆量之和
```

预览阶段库存不得变化。

## 7. 历史补录联调

入口：

```text
左侧导航
→ 历史补录
```

操作步骤：

1. 选择已有采购批次。
2. 选择一个或多个历史 CSV。
3. 设置日期、首锅时间或时间推断策略。
4. 点击解析预览。
5. 默认保持“影响库存”关闭。
6. 提交补录。

验收：

```text
status = completed
entry_mode = historical_backfill
inventory_effective = false
提交前后当前库存相同
曲线可以打开
批次可以进入复盘
```

## 8. 公开问卷联调

1. 选择一条 completed 烘焙批次。
2. 创建问卷。
3. 复制 `/#/eval/{shareCode}` 链接。
4. 退出登录或使用无痕窗口打开。
5. 页面应显示豆名和烘焙日期。
6. 填写综合喜好并提交。
7. 登录后台，在评价管理中查看结果。

以下两个接口必须保持匿名：


其余生豆、采购、烘焙、曲线、评价管理和复盘接口必须要求 Bearer token。

## 10. 常见故障

### Docker API 不可用

```text
failed to connect to the docker API
```

处理：启动 Docker Desktop，等待 Engine 完全启动，再执行 `docker info`。

### 前端提示网络连接失败

依次检查：

```powershell
docker compose ps
Invoke-WebRequest -UseBasicParsing http://127.0.0.1:8000/api/v1/openapi.json
Get-Content E:\Code_development\Coffee_bake_system\frontend\.env.local
```

### 返回 401

token 不存在或已失效。重新登录，不要把公开问卷误改成需要认证。

### commit 返回 409

常见原因：

- job 已提交。
- 文件哈希已存在。
- job 与 URL 的采购批次不匹配。
- 从错误入口提交了另一种 mode 的 job。

### 库存不足

批量提交会对采购批次加行锁，并在锁内重新计算库存。请降低有效行投豆量，或先补录采购/库存调整。

### 迁移后出现 InvalidCachedStatementError

停止并重启 FastAPI。迁移后不要继续复用迁移前建立的 asyncpg 连接池。

### PowerShell 中文显示乱码

Windows PowerShell 5.1 的控制台编码可能让 UTF-8 JSON 显示乱码。浏览器 `fetch` 不受该控制台显示问题影响。必要时先执行：

```powershell
[Console]::OutputEncoding = [System.Text.Encoding]::UTF8
```
