## 1. 项目目标

用户是一名咖啡烘焙师，需要制作一个咖啡烘焙数据分析系统。

核心目标：

- 记录咖啡生豆档案和采购批次。
- 记录烘焙批次和烘焙曲线。
- 支持不同豆种、不同烘焙批次之间的曲线对比。
- 支持风味维度评分。
- 支持外部人员通过问卷评价咖啡。
- 支持结合曲线、评价和个人复盘，总结烘焙变化对风味的影响。
- 后续可以扩展 RAG 和对话 Bot，但当前不是主线。

服务器条件：

- 阿里云服务器：2 核 4G。
- 用户在国内网络环境。
- 服务器部署建议：Docker Compose + FastAPI + PostgreSQL + Nginx。
- 当前优先级仍是本地前后端联调，不急于部署服务器。

## 2. 技术架构决策

后端：

- Python + FastAPI。
- PostgreSQL 作为主数据库。
- SQLAlchemy async。
- Alembic 管理数据库迁移。
- JWT 登录认证。
- CSV 文件上传后保存原文件，并解析为结构化曲线数据。

前端：

- Vue 3 + TypeScript + Vite。
- ECharts / vue-echarts 做曲线图。
- Pinia 管理状态。
- Vue Router 使用 Hash Router。

部署：

```text
用户浏览器
→ Nginx
   ├── Vue 静态文件
   ├── /api/ 反向代理 FastAPI
   └── /uploads/ 静态文件或受控文件访问
→ FastAPI
→ PostgreSQL
```

Docker 理解与决策：

- Docker 不是必须用于写代码，但推荐用于本地和服务器运行环境一致。
- 本地可以先只用 Docker 启动 PostgreSQL，前后端仍用开发模式运行。
- Docker 未安装好时，不影响代码层修改；只影响数据库容器、后端镜像和完整部署验收。

## 3. 已确认的数据对象

### 3.1 生豆档案 GreenBean

记录一款豆子的稳定信息：

- 名称。
- 豆种。
- 处理法。
- 产地 / 国家 / 地区 / 庄园。
- 海拔。
- 品牌。
- 产季。
- 豆商风味描述。

重要决策：

- 名称和豆种分离。
- 处理法应标准化，尽量使用固定词表，避免后续数据治理困难。

### 3.2 采购批次 PurchaseBatch

一款豆子可能反复采购，因此必须有采购批次。

字段：

- 生豆档案 ID。
- 采购时间。
- 总重量。
- 单价。
- 含水率。
- 批次号 / lot number。
- 供应商。
- 备注。

关系：

```text
GreenBean 1 - N PurchaseBatch
PurchaseBatch 1 - N RoastingBatch
```

库存规则：

```text
剩余库存 =
  采购批次总重量
  - 已完成且影响库存的烘焙批次投豆量
  + 手动库存调整
```

### 3.3 烘焙批次 RoastingBatch

用于记录一次烘焙。

字段：

- 采购批次 ID。
- 状态：planned / completed / voided。
- 计划时间。
- 实际烘焙时间。
- 计划投豆量。
- 实际投豆量。
- 出豆量。
- 脱水率。
- 总时长。
- 发展时间。
- 发展率。
- 烘焙度。
- 目标描述。
- 颜色标签。

已讨论需要新增：

- entry_mode。
- inventory_effective。
- roasted_at_source。
- bulk_import_group_id。
- source_note。

### 3.4 烘焙曲线 RoastingCurve / CurveFile

用户使用 Kaleido / 狙击手 M1 烘豆机 CSV。

CSV 中包含：

- BT。
- ET。
- RoR。
- 火力。
- 风门。
- 搅拌。
- 回温点。
- 转黄点。
- 一爆开始。
- 一爆结束。
- 出豆。
- 发展率等。

曲线展示要求：

- 默认同时显示温度、RoR、火力、风门。
- 不要把 RoR 单独放到很远的位置，应和 BT/ET 结合显示，便于烘焙判断。
- 支持 Artisan 风格浅色图表。
- 支持多条曲线对比。

### 3.5 问卷 Questionnaire 与评价 CuppingEvaluation

评价问卷从烘焙批次发起，不从评价管理页发起。

流程：

```text
烘焙批次
→ 创建问卷
→ 生成分享链接
→ 外部人员提交评价
→ 后台汇总评价
→ 复盘引用评价
```

评价字段：

- 干香。
- 湿香。
- 酸感强度。
- 甜感。
- 苦感强度。
- 回味。
- 综合喜好。
- 风味词。
- 冲煮方式。
- 饮用温度。
- 自由备注。

评分规则：

- 多数维度 1-5 分。
- 外部评价无需登录。
- 管理端查看评价需要登录。

### 3.6 复盘 BatchReview

复盘是对前面数据对象的评论和总结。

复盘内容：

- 个人复盘。
- 综合复盘。
- 烘焙建议。
- 下一次烘焙提醒。
- 下一次烘焙计划。

## 4. 已确认的主要操作流

### 4.1 生豆数据采集流

```text
新增或搜索生豆
→ 若不存在则创建生豆档案
→ 新增采购批次
→ 自动形成库存
→ 在树形结构中查看采购批次和烘焙记录
```

前端习惯：

- 录入和查看分开。
- 顶部一栏：搜索 + 新增按钮。
- 下方表格 / 树形结构查看。
- 生豆展开后第一级是采购记录，第二级是每次烘焙记录。

### 4.2 烘焙曲线分析流

```text
烘焙批次列表
→ 筛选豆种、采购批次、状态
→ 查看单条烘焙曲线
→ 选择两条或多条曲线对比
→ 分析事件点、阶段、火力、风门、RoR
```

### 4.3 评价管理流

评价管理只用于汇总分析，不用于发起问卷。

```text
烘焙批次创建问卷
→ 外部人员提交评价
→ 评价管理汇总查看
```

### 4.4 系统总览

总览需要包含：

- 按年份区分的烘豆频率。
- 已烘焙豆种数量。
- 已烘焙锅数。
- 产区分布。
- 最近一次烘焙列表。
- 待烘焙列表。
- 待完成事项。

待烘焙列表不一定需要复杂操作入口，可以标记是否完成，完成后计入已完成锅数。

## 7. 补录与多 CSV 功能最终决策

用户明确提出：

- 仍然需要历史补录。
- 但最近刚烘焙的一批 CSV，应迁移到建立烘焙批次流程中。
- 用户很少一次只烘一锅，通常一次采购后多次烘焙，之后批量上传 CSV。

最终设计：

```text
建立烘焙批次
├── 单锅计划
├── 单锅已完成录入
└── 批量 CSV 生成烘焙批次

历史数据补录
└── 用于旧数据归档，默认不影响库存
```

两个入口共用底层能力：

```text
CSV 解析
→ 时间推断
→ 预览确认
→ 创建 roasting_batch
→ 保存 curve_file
→ 生成 roasting_curve
→ 可选扣库存
→ 写入导入分组
```

### 7.1 多 CSV 生成烘焙批次

入口：

```text
采购批次详情
→ 建立烘焙批次
→ 批量 CSV 生成
```

默认规则：

- entry_mode = csv_bulk_import。
- status = completed。
- inventory_effective = true。
- 必须选择采购批次。
- 默认扣库存。

用途：

- 最近刚烘完，多锅 CSV 一次性上传。
- 自动生成多个 completed 烘焙批次。
- 自动按投豆量扣库存。
- 每个 CSV 对应一个烘焙批次和一条曲线。

### 7.2 历史数据补录

入口：

```text
系统设置 / 数据管理
→ 历史数据补录
```

默认规则：

- entry_mode = historical_backfill。
- status = completed。
- inventory_effective = false。
- 默认不影响当前库存。

用途：

- 旧 CSV 归档。
- 旧采购批次补建。
- 历史曲线进入系统用于对比、复盘和后续 RAG。

### 7.3 时间推断规则

不能完全依赖文件创建时间。

推荐优先级：

```text
1. CSV 内容中的烘焙时间，如果有
2. 文件名日期，例如 260530_9.csv
3. 浏览器上传时的 file.lastModified
4. 用户手动选择日期
5. 用户指定日期 + 上传顺序推断
```

必须记录：

```text
roasted_at
roasted_at_source
```
