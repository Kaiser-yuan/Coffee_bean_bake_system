# 咖啡烘焙系统：烘焙分析加载失败修改与验收指南

日期：2026-06-24  
适用版本：`master` / `5b45593` 之后的本地工作区  
问题页面：`http://127.0.0.1:5173/#/roasting`

## 1. 结论

本次“烘焙分析加载失败”不是 CSV 解析失败，也不是 PostgreSQL 数据丢失。

2026-06-24 23:08 至 23:09 的实际上传结果如下：

| 项目 | 结果 |
| --- | --- |
| 批量任务状态 | `committed` |
| 文件数 | 3 |
| 成功数 | 3 |
| 失败数 | 0 |
| `260530_9.csv` | `parsed`，257 个曲线点 |
| `260530_10.csv` | `parsed`，254 个曲线点 |
| `260530_11.csv` | `parsed`，282 个曲线点 |
| `GET /api/v1/roasting-batches` | 当前返回 200，共 9 条 |

当前浏览器重新加载后也能显示这 9 条记录。因此，截图中的错误是前端加载状态进入了不可恢复的失败状态，CSV 数据本身已经正确提交。

## 2. 根因与风险排序

### P0：失败的上下文 Promise 被永久缓存

文件：`frontend/src/services/greenBeanContextService.ts`

当前代码使用模块级变量：

```ts
let cache: RoastContext | null = null
let inflight: Promise<RoastContext> | null = null
```

请求成功时才执行：

```ts
cache = ctx
inflight = null
```

如果 `/green-beans/tree` 或 `/roasting-batches?page_size=500` 任意一次失败、超时或后端正好重启，`inflight` 会一直保存 rejected Promise。之后：

1. 页面点击“重试”。
2. `fetchRoastContext()` 发现 `inflight` 非空。
3. 再次返回同一个 rejected Promise。
4. 不会发起新的 HTTP 请求。
5. 页面持续显示“加载失败”，只能通过整页刷新或重启前端恢复。

`invalidateRoastContext()` 当前只清理 `cache`，也没有清理失败的 `inflight`，因此 CSV 提交后调用 invalidate 仍可能无法恢复。

### P0：非关键上下文失败导致整张列表不可见

文件：`frontend/src/views/RoastingBatchList.vue`

当前页面将两个任务放进同一个 `Promise.all`：

```ts
await Promise.all([
  store.fetchBatches(),
  fetchRoastContext().then(...),
])
```

烘焙批次列表本身只依赖 `store.fetchBatches()`。生豆上下文主要用于显示名称、豆种和筛选项，但上下文任意失败会让整个页面进入 ErrorState，已经成功取得的批次也不显示。

### P1：同一页面重复请求烘焙批次

当前一次进入烘焙分析会请求：

1. `store.fetchBatches()` -> `/roasting-batches`。
2. `fetchRoastContext()` -> `/green-beans/tree?archive_status=all`。
3. `fetchRoastContext()` 内部再次请求 `/roasting-batches?page=1&page_size=500`。

即烘焙批次接口被重复调用。两个响应只要有一个失败，整页都会失败。

### P1：生豆树存在库存 N+1 查询，当前已耗时 5.7 秒

本地实测：

| 请求 | 耗时 |
| --- | ---: |
| `/roasting-batches` | 184 ms |
| `/roasting-batches?page=1&page_size=500` | 161 ms |
| `/green-beans/tree?archive_status=all` | 5686 ms |

`backend/app/api/v1/green_beans.py` 在每个采购批次循环调用 `calculate_remaining_stock()`。该函数对每个采购批次执行采购记录、已烘消耗和库存调整查询。数据增长后会接近前端 15 秒超时。

本地 `.env` 还设置了 `DEBUG=true`，SQLAlchemy 会输出大量 SQL 和曲线 JSON，进一步放大批量上传后的瞬时延迟。

### P1：错误信息被丢弃

`RoastingBatchList.vue` 使用无参数 `catch`，只设置 `error = true`。页面只显示“加载失败”，无法区分：

- 后端未启动；
- 401 登录过期；
- 500 接口异常；
- 15 秒超时；
- 生豆上下文失败；
- 烘焙批次主列表失败。

这也是截图只能看到通用错误、无法直接判断原因的原因。

## 3. 修改一：确保失败 Promise 一定释放（P0）

修改 `frontend/src/services/greenBeanContextService.ts`，使用 `try/finally` 清理 inflight。推荐结构：

```ts
let cache: RoastContext | null = null
let inflight: Promise<RoastContext> | null = null
let generation = 0

export async function fetchRoastContext(force = false): Promise<RoastContext> {
  if (force) cache = null
  if (cache && !force) return cache
  if (inflight) return inflight

  const requestGeneration = generation
  const request = loadRoastContext()
  inflight = request

  try {
    const ctx = await request
    if (requestGeneration === generation) cache = ctx
    return ctx
  } finally {
    if (inflight === request) inflight = null
  }
}

export function invalidateRoastContext() {
  cache = null
  generation += 1
}
```

`loadRoastContext()` 内保留真实 API 与 demo mode 的加载逻辑。关键要求：成功、HTTP 错误、网络错误和超时都必须进入 `finally`。

不要简单地在 `invalidateRoastContext()` 中无条件把正在运行的 Promise 设为 null 后仍允许旧请求写回 cache；使用 generation 防止旧响应覆盖新状态。

## 4. 修改二：主列表与辅助上下文分级处理（P0）

修改 `frontend/src/views/RoastingBatchList.vue`：

1. `store.fetchBatches()` 是主请求，失败时显示整页错误。
2. 生豆上下文是辅助请求，失败时保留批次列表，只显示黄色提示。
3. 点击“重试”必须重新发请求，不得复用 rejected Promise。

推荐结构：

```ts
const loadError = ref('')
const contextWarning = ref('')

async function fetchBatches() {
  loading.value = true
  error.value = false
  loadError.value = ''
  contextWarning.value = ''

  const [batchResult, contextResult] = await Promise.allSettled([
    store.fetchBatches(),
    fetchRoastContext(),
  ])

  if (batchResult.status === 'rejected') {
    error.value = true
    loadError.value = toReadableError(batchResult.reason)
  }

  if (contextResult.status === 'fulfilled') {
    roastContext.value = contextResult.value
  } else {
    contextWarning.value = `批次已加载，但生豆信息加载失败：${toReadableError(contextResult.reason)}`
  }

  loading.value = false
}
```

模板改为：

```vue
<ErrorState
  v-else-if="error"
  :message="loadError"
  :retry="true"
  @retry="fetchBatches"
/>
<p v-if="contextWarning" class="warning-banner">{{ contextWarning }}</p>
```

辅助上下文失败时，生豆名称可以暂时显示后端列表响应中的 `green_bean_name`，不要把整张表隐藏。

## 5. 修改三：消除重复批次请求（P1）

推荐拆分上下文服务：

```ts
fetchGreenBeanTreeContext()
fetchRoastContext()
```

烘焙分析页已经通过 store 获取批次，因此只需并行请求：

```ts
await Promise.all([
  store.fetchBatches(),
  fetchGreenBeanTreeContext(),
])
```

不要再由 `fetchRoastContext()` 为该页面请求第二次 `page_size=500` 的烘焙批次。

其他确实需要完整 `RoastContext` 的页面可以继续调用组合方法，但底层缓存必须分别管理，不能让 tree 请求失败污染 batch 请求缓存。

## 6. 修改四：批量计算库存，移除 N+1（P1）

后端新增批量方法：

```py
async def calculate_remaining_stocks(
    db: AsyncSession,
    purchase_batch_ids: list[str],
) -> dict[str, int]:
    ...
```

使用两个聚合子查询：

1. 按 `purchase_batch_id` 汇总 completed 且 `inventory_effective=true` 的实际投豆量。
2. 按 `purchase_batch_id` 汇总库存调整量。
3. 与 `purchase_batches` 一次连接，计算：

```text
COALESCE(opening_stock_grams, total_weight_grams)
- COALESCE(roast_consumed, 0)
+ COALESCE(adjustment_sum, 0)
```

`get_green_bean_tree()` 先收集全部采购批次 ID，一次取得 `remaining_by_pb`，构造响应时只查字典：

```py
remaining_by_pb = await calculate_remaining_stocks(db, pb_ids)
remaining = remaining_by_pb.get(pb.id, 0)
```

验收时不应再看到每个采购批次重复执行三组库存 SQL。

## 7. 修改五：保留可诊断错误（P1）

前端增加统一方法：

```ts
function toReadableError(error: unknown): string {
  if (error instanceof ApiError) {
    const request = error.requestId ? `，request_id=${error.requestId}` : ''
    return `${error.message}${request}`
  }
  return '未知加载错误'
}
```

页面 catch 中至少执行：

```ts
console.error('Failed to load roasting batches', error)
```

后端建议：

- 开发联调需要 SQL 时临时开启 `DEBUG=true`。
- 正常人工验收改为 `DEBUG=false`，避免打印大段曲线 JSON。
- 应用级日志保留路径、状态码、处理耗时和 request_id。

## 9. 自动化测试

### 9.1 上下文失败后可恢复

为 `greenBeanContextService` 增加测试：

1. 第一次 mock `/green-beans/tree` 返回 reject。
2. 第一次 `fetchRoastContext()` 应 reject。
3. 第二次 mock 返回正常数据。
4. 第二次调用必须重新发送请求并成功。
5. 断言请求次数为 2。

### 9.2 invalidate 期间旧请求不得污染缓存

1. 启动一个延迟返回的上下文请求 A。
2. 调用 `invalidateRoastContext()`。
3. 请求 B 返回新数据。
4. 最后让请求 A 返回旧数据。
5. 再次读取必须得到 B，不得得到 A。

### 9.3 页面部分降级

1. mock `/roasting-batches` 返回 9 条。
2. mock `/green-beans/tree` 返回 500。
3. 表格仍显示 9 条。
4. 页面显示“生豆信息加载失败”警告。
5. 不显示整页“加载失败”。

### 9.4 请求去重

进入烘焙分析一次，断言：

- `/roasting-batches` 只请求一次。
- `/green-beans/tree` 只请求一次。
- 不再额外请求 `/roasting-batches?page_size=500`。
