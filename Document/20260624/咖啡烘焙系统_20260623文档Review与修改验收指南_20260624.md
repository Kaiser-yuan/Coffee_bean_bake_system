## 1.1 P0：修复单锅曲线上传后不显示

### 已定位根因

`frontend/src/views/CurveAnalysis.vue` 的 `fetchCurves()` 存在确定的渲染时序错误：

1. 函数先设置 `loading.value = true`。
2. 页面模板使用 `v-if="loading"`，因此曲线容器被移出 DOM。
3. 数据请求完成后，在 `loading` 仍为 true 时执行 `await nextTick()` 和 `renderChart()`。
4. `renderChart()` 检测不到 `mainChartRef`，直接返回。
5. `finally` 才设置 `loading.value = false`，曲线容器虽然重新出现，但没有再次执行绘图。

重新上传或切换批次时还有第二个问题：旧 ECharts 实例仍绑定在已经销毁的旧 DOM 容器上。即使再次调用 `setOption()`，也可能继续向不可见的旧容器绘图。

### 前端修改

修改 `CurveAnalysis.vue`：

- 引入 `onBeforeUnmount`。
- 每次重新获取曲线前，销毁旧 ECharts 实例并将变量置为 null。
- 数据请求阶段只更新 `curves`，不要立即绘图。
- 在 `finally` 中先结束 loading。
- loading 结束后再执行 `await nextTick()`，确认新图表容器已经挂载。
- 最后调用 `renderChart()`。
- 页面卸载时销毁 ECharts 实例，避免残留监听器和内存泄漏。

推荐结构：

```ts
async function fetchCurves() {
  chart?.dispose()
  chart = null

  loading.value = true
  error.value = false
  let shouldRender = false

  try {
    // 保留现有曲线查询、排序和问卷状态加载逻辑
    shouldRender = true
  } catch (e) {
    error.value = true
    loadError.value = e instanceof ApiError ? e.message : '曲线加载失败'
  } finally {
    loading.value = false
  }

  if (shouldRender) {
    await nextTick()
    renderChart()
  }
}

onBeforeUnmount(() => {
  chart?.dispose()
  chart = null
})
```

建议同时增加窗口 resize 处理，使侧栏展开、窗口缩放后图表自动调用 `chart.resize()`；对应监听器也必须在卸载时清理。

### 后端与环境前置检查

前端修复前先确认真实 API 模式和数据库版本，否则“上传失败”可能被误判为“绘图失败”。

`frontend/.env.local` 至少包含：

```env
VITE_DEMO_MODE=false
VITE_API_BASE_URL=http://localhost:8000/api/v1
```

后端执行：

```bash
uv run alembic upgrade head
uv run alembic current
```

必须包含 `0003_widen_curve_version` 及其后续迁移。该迁移把 `roasting_curves.calculation_version` 从 16 位扩展到 32 位，否则保存 `curve-analysis-v1` 时会失败。迁移后必须重启 FastAPI 服务。

### 分层定位方法

1. `POST /api/v1/roasting-batches/{batch_id}/curve-files` 返回 201，且 `parse_status=parsed`、`data_points>0`。
2. `GET /api/v1/roasting-batches/{batch_id}/curve` 返回 200，且 `points.length>0`。
3. 前端 `curveService.toPoint()` 映射后，至少 BT 或 ET 存在有效数值。
4. 图表容器宽度大于 0、高度为 480px，ECharts 实例绑定当前容器。

判断规则：

- 上传接口返回 failed：解析、数据库迁移或文件存储问题。
- 上传成功但查询 404：曲线未激活或事务未正确保存。
- 查询成功但 points 为空：后端存储或响应构造问题。
- points 有数据但页面空白：前端渲染时序、字段映射或容器问题。

## 2. 修改一：产季改为可选年份

### 当前问题

GreenBeanManagement.vue 使用文本框输入 harvest_season，用户可能录入 2025、2025产季、25/26 等不同格式，后续筛选和统计困难。

### 数据策略

第一阶段可以继续复用数据库 harvest_season 字符串字段，前端和后端统一只保存四位年份，避免立即改表。

推荐规则：

- 可为空。
- 有值时必须是四位年份。
- 允许范围为 1900 到当前年份加 1。
- 默认选项为当前年份、前一年和下一年。
- 下拉列表建议提供最近 15 至 20 年。

后续若需要跨年产季，再单独设计 harvest_start_year 和 harvest_end_year，不要继续混用自由文本。

### 后端修改

在 GreenBeanWithFirstPurchaseRequest 和后续新增的 GreenBeanUpdateRequest 中增加 harvest_season 校验。

非法示例：

- 25年。
- 2025产季。
- abc。
- 1800。

这些输入应返回 422。

### 前端修改

把文本框改为 select：

- 未填写。
- 2027。
- 2026。
- 2025。
- 向前动态生成若干年份。

显示时统一为“2026 产季”，传给后端时只传“2026”。

### 兼容旧数据

增加一次数据清理脚本，将 2025产季、2025 产季、2025新产季等可识别值归一为 2025；无法识别的值输出清单，人工确认。

### 验收

- [ ] 生豆录入页不再允许自由输入产季。
- [ ] 产季可以为空。
- [ ] 新增数据只保存四位年份。
- [ ] 老数据规范化后仍能显示。
- [ ] 按年份筛选和统计结果一致。

- 删除和归档必须二次确认。

## 6. 修改五：烘焙批次删除与作废规则

### 当前问题

页面操作菜单已有“作废批次”，但没有明确删除按钮。对 completed 批次直接物理删除会破坏库存流水、曲线、问卷、评价和复盘关系。

### 推荐规则

#### 待烘计划

- 没有曲线、评价和复盘时允许物理删除。
- 新增 DELETE /api/v1/roasting-batches/{batch_id}。

#### 已完成批次

- 不提供普通物理删除。
- 使用现有 void 操作作废。
- 作废时按 inventory_effective 恢复库存。
- 保留曲线、评价和复盘作为历史。

#### 误导入的历史/批量数据

第一阶段也使用作废。

如确需永久删除，后续增加管理员专用 purge：

- 明确列出会删除的曲线、文件、问卷、评价、复盘和库存流水。
- 二次输入批次号确认。
- 数据库事务和文件系统补偿同时处理。

### 前端修改

- planned 显示“删除计划”。
- completed 显示“作废批次”。
- voided 显示“已作废”，默认不进入统计。
- 不要把所有情况统一叫“删除”。

### 验收

- [ ] 删除待烘计划不会影响库存。
- [ ] completed 批次无法普通物理删除。
- [ ] 作废 completed 批次后库存正确返还。
- [ ] archive-only 批次作废不产生库存返还。
- [ ] 曲线和评价历史仍可查看。
- [ ] 重复删除或作废返回 409。

## 7. 修改六：修复曲线对比加载与对齐

### 已定位的问题

1. compare API 返回 aligned_seconds，但前端 toPoint 忽略该字段，图表仍使用 elapsed_seconds。
2. 前端 compareCurves 固定 align_by=charge，用户无法选择其他对齐方式。
3. 部分请求批次没有曲线时，后端只在有效曲线少于 2 条时报错；请求 3 条但仅返回 2 条时可能静默遗漏。
4. 某条曲线缺少 charge 事件时，整个对比失败。
5. 页面错误提示不足，用户只看到曲线无法加载。

### 后端修改

compare endpoint 必须严格检查：

- 请求批次 ID 去重。
- 返回曲线数量必须等于请求数量。
- 缺少曲线时返回 missing_batch_ids。
- 对齐事件缺失时返回具体 batch_id 和 event_type。

支持 align_by：

- none，使用原始时间。
- charge。
- yellowing。
- first_crack_start。
- drop。

对齐事件缺失策略：

- strict=true 时返回 422。
- strict=false 时该批次退回原始时间并返回 warning。

### 前端修改

CurvePoint 增加 alignedElapsedSeconds，或者比较模式下直接：

    elapsedSeconds = aligned_seconds ?? elapsed_seconds

曲线对比页增加对齐选择控件：

- 原始时间。
- 入豆。
- 转黄。
- 一爆开始。
- 出豆。

接口错误要显示：

- 哪些批次没有曲线。
- 哪些批次缺少对齐事件。
- 哪个 CSV 解析失败。

### 图表检查

- BT、ET、RoR、火力、风门、滚筒数据分别映射正确。
- step 类型控制曲线使用阶梯线。
- 对比颜色按批次稳定。
- 同一批次在图例、列表和曲线中使用相同颜色。

### 验收

- [ ] 选择两条真实 CSV 曲线后图表非空。
- [ ] 两条曲线都出现在图例。
- [ ] charge 对齐时入豆事件位于 0 秒。
- [ ] 原始时间模式不修改时间轴。
- [ ] 切换一爆对齐后图表重新加载。
- [ ] 缺曲线时明确显示批次号。
- [ ] 缺入豆事件时可以改用原始时间。
- [ ] 请求三条、缺一条时不会静默只显示两条。
