# 咖啡烘焙系统：后端旧进程与请求超时修改验收指南

日期：2026-06-24  
问题页面：`http://127.0.0.1:5173/#/roasting`  
结论：运行版本错配，不是本轮归档过滤代码失败

## 1. 本次结论

本轮修改已经正确提交到本地 Git：

```text
30fc5a7 fix: roasting list defaults to active green beans
c8f72f3 fix: roasting analysis load failure
```

但问题发生时，端口 8000 上仍运行修改前启动的 Uvicorn 进程，启动命令是：

```text
uvicorn app.main:app --host 0.0.0.0 --port 8000
```

该命令没有 `--reload`。修改 Python 文件和 Git 提交不会自动更新已经加载到内存中的 FastAPI 应用。

因此当时实际状态是：

```text
前端：最新代码，发送 bean_archive_status=active
后端：旧代码，不认识 bean_archive_status
数据库：正常
```

这会造成：

1. 前端显示新的“在用生豆批次”筛选。
2. 旧后端忽略陌生查询参数，仍返回全部 9 条。
3. 旧后端仍执行逐采购批次库存查询。
4. 生豆树请求耗时波动，叠加 SQL 调试日志或批量上传时可能超过前端 15 秒。
5. 前端显示“请求超时”，看起来像新代码无数据。

重启后端加载当前 HEAD 后，页面已恢复为 3 条。

## 4. 修改一：增加统一开发启动脚本（P0）

新增：

```text
scripts/dev-backend.ps1
```

职责：

1. 检查 Docker PostgreSQL 是否 healthy。
2. 检查 8000 端口是否已被占用。
3. 如果是本项目旧 Uvicorn，明确提示并停止。
4. 执行 Alembic current/upgrade。
5. 使用 `--reload` 启动。
6. 输出当前 Git SHA 和 API 地址。

推荐脚本核心：

```powershell
$ErrorActionPreference = 'Stop'
$repo = 'E:\Code_development\Coffee_bake_system'
$backend = Join-Path $repo 'backend'
$uv = 'C:\Users\Kaiser\AppData\Local\Programs\Python\Python312\Scripts\uv.exe'

Set-Location $backend
docker compose up -d db
& $uv run alembic upgrade head

$sha = git -C $repo rev-parse --short HEAD
Write-Host "Starting backend from commit $sha"

& $uv run uvicorn app.main:app --host 127.0.0.1 --port 8000 --reload
```

不要再用不带 `--reload` 的裸 Uvicorn 命令进行日常开发。

生产环境仍然不能使用 `--reload`。生产部署必须通过重新构建/重启容器加载新版本。

## 5. 修改二：增加运行版本探针（P0）

当前根接口只返回固定版本 `0.1.0`，无法判断进程加载的是哪个提交。

新增接口：

```text
GET /api/v1/meta
```

推荐响应：

```json
{
  "service": "coffee-roast-api",
  "app_version": "0.3.0",
  "git_sha": "30fc5a7",
  "api_contract_version": "2026-06-24.2",
  "features": [
    "bean_archive_status",
    "batch_stock_aggregation",
    "curve_aligned_seconds"
  ]
}
```

实现要求：

- `git_sha` 通过启动环境变量 `APP_GIT_SHA` 注入。
- 不要在每次请求中运行 git 命令。
- Docker 构建时写入镜像标签或环境变量。
- `/meta` 不返回密码、数据库 URL、secret 或内部路径。

前端启动真实 API 模式时请求 `/meta`：

```ts
if (!meta.features.includes('bean_archive_status')) {
  throw new Error('后端版本过旧，请重启后端服务')
}
```

版本不匹配时应显示明确提示，不能继续显示“请求超时”。

## 6. 修改三：增加就绪检查（P1）

新增：

```text
GET /api/v1/health/ready
```

至少检查：

- 应用进程可响应；
- PostgreSQL `SELECT 1` 成功；
- Alembic revision 为 head；
- 上传目录可写。

推荐响应：

```json
{
  "status": "ready",
  "database": "ok",
  "migration": "0006_green_bean_archive",
  "upload_storage": "ok",
  "git_sha": "30fc5a7"
}
```

前端只需用它判断服务可用性，不要把 health 请求当业务数据请求。

## 7. 修改四：拆分 DEBUG 与 SQL_ECHO（P1）

当前：

```py
engine = create_async_engine(
    settings.database_url,
    echo=settings.debug,
)
```

本地 `.env` 为：

```text
DEBUG=true
```

这会把每条 SQL 及大型曲线 JSON 写入日志。批量上传和树查询时会明显拖慢响应。

增加独立配置：

```py
debug: bool = True
sql_echo: bool = False
```

数据库引擎改为：

```py
echo=settings.sql_echo
```

本地 `.env`：

```text
DEBUG=true
SQL_ECHO=false
```

只有定位 SQL 时临时设置 `SQL_ECHO=true`，排查结束后恢复 false。

禁止用扩大前端超时来掩盖 SQL 日志和旧进程问题。

## 8. 修改五：前端区分超时、版本过旧和服务未启动（P1）

推荐错误优先级：

1. `/meta` 无法连接：后端未启动。
2. `/meta` 能连接但 feature 缺失：后端进程版本过旧。
3. `/health/ready` 非 ready：数据库、迁移或存储未就绪。
4. 业务请求 AbortError：业务请求超时。
5. HTTP 4xx/5xx：显示后端 message/request_id。

示例提示：

```text
后端进程版本过旧：当前不支持 bean_archive_status。
请重启 8000 端口后端，而不是重新上传 CSV。
```

前端保留 15 秒默认超时。当前最新后端请求均远低于 15 秒，无需先增加到 30 或 60 秒。

## 9. 修改六：后端改动后的强制检查流程（P0）

每次修改以下目录后执行：

```text
backend/app/
backend/alembic/
backend/.env
```

检查顺序：

1. 观察 Uvicorn 是否输出 reload 日志。
2. `.env` 和依赖变化时，即使有 reload 也要完整重启。
3. 执行 OpenAPI 契约检查。
4. 执行业务接口 smoke test。
5. 最后刷新前端。

归档过滤契约检查：

```powershell
$spec = Invoke-RestMethod http://127.0.0.1:8000/api/v1/openapi.json
$params = $spec.paths.'/api/v1/roasting-batches'.get.parameters.name
$params -contains 'bean_archive_status'
```

必须输出：

```text
True
```

如果输出 False，立即停止前端验收，先重启后端。
