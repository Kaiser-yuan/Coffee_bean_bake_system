# 咖啡烘焙系统：本地串行功能测试指南

> 日期：2026-06-24  
> 目标：用一条连续操作路线，快速确认本地前端、后端、数据库和主要业务功能是否正常。

## 测试前准备

准备以下测试材料：

- 两份未导入过的正常 Kaleido M1 CSV，例如 `260530_9.csv`、`260530_10.csv`。
- 一份未导入过的旧 CSV，用于历史补录；没有时可以暂时跳过历史补录。
- 测试豆名称：`本地测试豆-0624`。
- 第一次采购 1000g，第二次采购 500g。
- 每次烘焙投豆量统一填写 100g。

测试过程中只要某一步失败，先停在当前步骤记录错误，不要继续录入更多数据。

## 1. 启动后端和数据库

在 `backend` 目录执行：

```powershell
docker compose up -d --build
docker compose exec backend alembic upgrade head
```

打开：

- 后端文档：`http://localhost:8000/api/v1/docs`
- 后端健康检查：`http://localhost:8000/health`

通过标准：两个地址可以打开，Docker 中 `db` 和 `backend` 都处于运行状态。

## 2. 启动前端

确认 `frontend/.env.local` 内容：

```env
VITE_API_BASE_URL=http://localhost:8000/api/v1
VITE_DEMO_MODE=false
VITE_BASE_PATH=/
```

在 `frontend` 目录执行：

```powershell
npm install
npm run dev
```

打开 `http://localhost:5173`。

通过标准：自动进入登录页，不显示演示数据。

## 3. 登录系统

使用已经创建的管理员账号登录。

如果数据库是第一次启动，在 Swagger 中调用 `POST /api/v1/auth/register` 创建第一个管理员，再返回前端登录。

通过标准：登录后进入“系统总览”，刷新页面后仍保持登录。

## 4. 检查系统设置

进入“系统设置”，查看处理法、风味词、评价者类型等标准词表。

尝试新增一个测试词条，再停用或恢复它。

通过标准：保存后刷新页面仍然存在，表单中的下拉选项能读取这些词条。

## 5. 新建生豆和第一次采购

进入“生豆管理”，点击新增，填写：

- 名称：本地测试豆-0624。
- 豆种、处理法、产地、海拔、品牌、产季：按实际测试内容填写。
- 第一次采购重量：1000g。
- 购买时间和价格：填写任意合理值。

通过标准：列表出现该生豆，展开后能看到第一次采购记录，剩余库存为 1000g。

## 6. 编辑生豆并新增第二次采购

先修改一次生豆名称或风味描述并保存，然后刷新页面。

再给同一款生豆新增第二次采购，采购重量填写 500g。

通过标准：

- 修改后的生豆信息没有丢失。
- 同一生豆下有两条采购记录。
- 总剩余库存为 1500g。
- 生豆档案和采购批次没有被错误合并成同一条记录。

## 7. 创建并删除一个待烘计划

从第一条采购记录创建一个待烘计划，投豆量填写 100g，但先不要上传 CSV。

然后删除这个待烘计划。

通过标准：计划可以删除，库存始终保持 1500g，没有发生扣减。

## 8. 批量上传两份 CSV

在第一条采购记录下点击“批量 CSV”，同时选择两份正常 CSV。

在预览页确认：

- 每个 CSV 被识别为一条独立烘焙批次。
- 烘焙时间从 CSV 中自动取得。
- 总时长和一爆时间显示为“分:秒”。
- 每锅投豆量填写 100g。
- 两锅都设置为影响库存。

确认导入。

通过标准：生成两个已完成烘焙批次，第一条采购记录减少 200g，全豆总剩余库存变为 1300g。

## 9. 查看第一锅单锅曲线

进入“烘焙分析”，打开第一锅曲线。

通过标准：

- 页面无需手动刷新即可出现曲线。
- BT、ET、RoR、火力、风门和滚筒默认一起显示。
- 点击图例可以隐藏和恢复某条线。
- 总时长、一爆和发展率与 CSV 基本一致。
- 页面没有一直加载、空白图表或控制台 ECharts 错误。

如果接口返回了非空 `points` 但页面仍为空，优先检查 `CurveAnalysis.vue` 的 loading、nextTick 和 ECharts 实例销毁顺序。

## 10. 对比两锅曲线

返回“烘焙分析”，勾选刚导入的两锅并进入对比。

依次尝试原始时间、入豆对齐和一爆对齐。

通过标准：

- 两条曲线都出现在图例和图表中。
- 切换对齐方式后图表会重新计算。
- 入豆对齐时入豆点位于 0 秒。
- 不会只显示其中一锅，也不会把两锅颜色混在一起。

## 11. 重新上传单锅曲线

回到第一锅详情，重新上传另一份可正常解析的 CSV。

通过标准：新曲线替换旧曲线，页面立即重新绘制；返回列表后显示当前生效的 CSV 文件名、上传时间和解析状态，库存仍为 1300g。

## 12. 测试烘焙列表筛选

在“烘焙分析”中分别按豆种、日期、状态和数据完整度筛选。

通过标准：筛选结果正确；两锅都能查看曲线；缺评价、缺复盘等提示与实际状态一致。

## 13. 发起评价问卷

进入第一锅曲线详情，点击“发起评价问卷”。

复制公开链接，并在浏览器无痕窗口中打开。

通过标准：未登录也能打开问卷，页面显示对应咖啡和烘焙批次，不需要从问卷里重新选择批次。

## 14. 提交一份外部评价

在公开问卷中填写评价者信息、干香、湿香、酸、甜、苦、回味等分数，并填写一条自由风味描述后提交。

通过标准：只能成功提交一次，提交后显示感谢页面，刷新不会重复写入。

## 15. 查看评价汇总

登录后台进入“评价管理”，打开刚才的问卷详情。

通过标准：

- 总评价数增加 1。
- 各风味维度显示刚提交的分数。
- 自由风味描述可以查看。
- 评价管理只负责汇总、查看和关闭，不出现脱离烘焙批次的新建问卷入口。

测试完成后关闭问卷，再打开公开链接。

通过标准：问卷显示已关闭，不能继续提交。

## 16. 完成批次复盘

从第一锅进入“批次复盘”。

先填写个人初始复盘并保存，再查看评价摘要，最后填写综合复盘和下一锅提醒。

通过标准：

- 刷新后个人复盘仍存在。
- 评价摘要来自当前烘焙批次。
- 综合复盘可以保存和更新。
- 下一锅提醒与当前批次正确关联。

## 17. 测试历史补录

进入“历史数据补录”，选择一份从未导入过的旧 CSV。

选择已有生豆和采购批次，保持“影响库存”关闭后提交。

通过标准：系统生成历史烘焙批次和曲线，但当前库存仍为 1300g，并明确显示为历史补录或不影响库存。

## 18. 作废一个已完成批次

在“烘焙分析”中作废第二锅正常烘焙批次。

通过标准：

- 批次状态变为已作废。
- 该批次不再进入正常统计。
- 因为该锅原本扣除了 100g，库存从 1300g 恢复为 1400g。
- 原曲线仍可作为历史记录查看。
- 重复作废不会再次增加库存。

## 19. 检查系统总览

回到“系统总览”，检查年度烘焙频率、豆种分布、产区分布、已完成锅数、最近烘焙列表和待烘列表。

通过标准：正常批次、历史补录批次和已作废批次按照系统规则统计，不出现重复锅数，刚才的库存和批次状态能对应起来。

## 20. 测试生豆归档与恢复

归档“本地测试豆-0624”。

通过标准：默认列表不再显示，但曲线、评价和复盘历史没有删除。

切换到归档数据并恢复。

通过标准：生豆重新出现，两次采购和所有历史关系仍然完整。

## 21. 最后做一次刷新检查

依次刷新以下页面：

1. 系统总览。
2. 生豆管理。
3. 烘焙分析。
4. 第一锅曲线。
5. 评价管理。
6. 批次复盘。

通过标准：没有数据消失、页面空白、无限加载或需要重新录入的情况。

## 简单记录方式

每一步只记录以下三种结果：

- `通过`：结果符合指南。
- `失败`：记录步骤号、页面、错误提示和一张截图。
- `跳过`：当前功能尚未开发完成或没有合适测试文件。

建议当晚优先保证第 1 到第 16 步通过。第 17 步以后属于补录、作废、统计和归档回归，可以在还有耐心时继续。
