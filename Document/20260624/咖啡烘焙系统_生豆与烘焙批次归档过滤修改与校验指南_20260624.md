# 咖啡烘焙系统：生豆与烘焙批次归档过滤修改与校验指南

日期：2026-06-24  
问题页面：`http://127.0.0.1:5173/#/roasting`  
问题性质：列表范围错误，不是外键关系损坏，也不是 CSV 重复解析

## 1. 结论

当前数据库的一对多关系完整，没有发现孤儿记录或同一烘焙批次存在多条生效曲线。

用户看到“2 个生豆、3 个批次，但烘焙分析出现 9 条”的真实原因是：

1. 生豆管理默认只显示 `is_archived=false` 的 2 个在用生豆。
2. 数据库实际还有 9 个已归档生豆。
3. 这 9 个归档生豆下面保留了 6 个历史烘焙批次。
4. `/api/v1/roasting-batches` 没有按生豆归档状态过滤，默认返回在用和归档数据。
5. 烘焙分析因此显示 3 个当前批次 + 6 个归档历史批次，共 9 条。

所以用户对默认页面应显示 3 条的判断是正确的；需要修复的是“默认列表范围”，不能删除或合并合法的历史关系。

## 3. 正确的数据关系

```text
GreenBean 生豆档案
  1 ── N PurchaseBatch 采购批次
           1 ── N RoastingBatch 烘焙批次
                    1 ── N CurveFile 原始曲线文件
                    1 ── 0..1 RoastingCurve 当前生效曲线
```

说明：

- 同一生豆可以多次采购。
- 同一采购批次可以烘焙多锅。
- 每一锅烘焙只能有一条“当前生效曲线”。
- 重新上传 CSV 时可以保留多个 CurveFile 作为文件历史，但 RoastingCurve 仍只有一条，并指向当前文件。
- 生豆归档只改变可见范围，不删除采购、烘焙、曲线、评价和复盘历史。

烘焙分析表格中的每一行是一个 `RoastingBatch`，不是一个独立生豆，也不是 CurveFile 历史版本。

## 4. 根因代码

### 4.1 后端列表没有生豆归档过滤

文件：`backend/app/api/v1/roasting_batches.py`

当前列表参数只有：

```py
status
purchase_batch_id
search
has_curve
page
page_size
```

缺少 `bean_archive_status`。

文件：`backend/app/repositories/roasting_batches.py`

`get_list()` 虽然加载了：

```py
RoastingBatch.purchase_batch -> PurchaseBatch.green_bean
```

但查询没有：

```py
GreenBean.is_archived.is_(False)
```

所以默认会返回所有烘焙批次。

### 4.2 前端默认请求没有指定数据范围

文件：`frontend/src/services/roastingBatchService.ts`

当前调用：

```ts
listRoastingBatches({ has_curve: undefined })
```

没有传生豆归档状态。

### 4.3 前端上下文主动加载全部生豆

文件：`frontend/src/services/greenBeanContextService.ts`

```ts
getGreenBeanTree({ archive_status: 'all' })
```

这里加载全部数据本身没有错，因为查看历史曲线时仍需要解析归档生豆名称。但烘焙主列表没有单独过滤，于是历史数据也进入默认页面。

### 4.4 响应没有归档状态字段

`RoastingBatchResponse` 有 `green_bean_name`，但没有 `green_bean_is_archived`。前端即使收到归档批次，也无法直接显示“已归档生豆”标识，只能通过上下文间接推断。

## 5. 目标产品规则

### 5.1 默认烘焙分析

- 只显示 `GreenBean.is_archived=false` 对应的烘焙批次。
- 按当前数据库应显示 3 条。
- 测试豆1没有烘焙批次，因此不会产生空行。
- 测试豆2显示三条 CSV 批次。

### 5.2 归档历史

- 不物理删除归档生豆的烘焙批次和曲线。
- 增加“生豆范围”筛选：
  - 在用生豆，默认；
  - 已归档生豆；
  - 全部历史。
- 已归档批次必须显示“已归档生豆”标签。
- 仍允许进入曲线、评价和复盘详情。
- 不允许基于已归档生豆创建新的待烘计划或上传新的生产批次，除非先恢复生豆。

### 5.3 历史补录

- `entry_mode=historical_backfill` 与 `GreenBean.is_archived` 是两个不同维度。
- 历史补录可以属于在用或归档生豆。
- “仅归档”标签由 `inventory_effective=false` 决定。
- 生豆归档筛选不能替代 entry mode 和库存影响标签。

## 6. 修改一：后端增加归档范围参数（P0）

修改 `backend/app/api/v1/roasting_batches.py`：

```py
from typing import Literal

@router.get("")
async def list_roasting_batches(
    status: str | None = None,
    purchase_batch_id: str | None = None,
    search: str | None = None,
    has_curve: bool | None = None,
    bean_archive_status: Literal["active", "archived", "all"] = "active",
    page: int = 1,
    page_size: int = 20,
    ...
):
    items, total = await repo.get_list(
        ...,
        bean_archive_status=bean_archive_status,
    )
```

默认必须是 `active`。需要全历史数据的调用方必须显式传 `all`，避免未来新增页面再次泄漏归档数据。

## 7. 修改二：仓储层按归档状态过滤（P0）

修改 `backend/app/repositories/roasting_batches.py`：

```py
async def get_list(
    ...,
    bean_archive_status: str = "active",
) -> tuple[list[RoastingBatch], int]:
    stmt = (
        select(RoastingBatch)
        .join(RoastingBatch.purchase_batch)
        .join(PurchaseBatch.green_bean)
        .options(...)
    )

    if bean_archive_status == "active":
        stmt = stmt.where(GreenBean.is_archived.is_(False))
    elif bean_archive_status == "archived":
        stmt = stmt.where(GreenBean.is_archived.is_(True))
```

要求：

- count 查询必须基于应用归档过滤后的 stmt。
- 分页前过滤，不能先取 20 条再由前端剔除。
- `purchase_batch_id`、status、has_curve 和搜索条件必须继续组合生效。
- `all` 不增加归档条件。

顺便修复：当前 `search` 参数传入 repository 后没有实际应用。建议搜索生豆名称时增加：

```py
if search:
    stmt = stmt.where(GreenBean.name.ilike(f"%{search}%"))
```

## 8. 修改三：响应明确返回归档状态（P1）

在 `RoastingBatchResponse` 增加：

```py
green_bean_is_archived: bool = False
```

在 `_to_batch_response()` 中从关联生豆赋值：

```py
green_bean_is_archived=(
    batch.purchase_batch.green_bean.is_archived
    if batch.purchase_batch and batch.purchase_batch.green_bean
    else False
)
```

同步修改：

- `frontend/src/api/types.ts`
- `frontend/src/types/index.ts`
- `frontend/src/adapters/roastingBatch.ts`

领域模型建议字段：

```ts
greenBeanIsArchived?: boolean
```

不要只依赖全局 GreenBean Context 推断归档状态，因为上下文加载失败时主列表仍应正确标识。

## 9. 修改四：前端筛选与请求契约（P0）

### 9.1 API 参数

修改 `frontend/src/api/roastingBatches.ts`：

```ts
export interface ListRoastingBatchesParams {
  ...
  bean_archive_status?: 'active' | 'archived' | 'all'
}
```

构造查询参数：

```ts
if (params.bean_archive_status) {
  qs.set('bean_archive_status', params.bean_archive_status)
}
```

### 9.2 Service 和 Store

让 `fetchRoastingBatches()`、store 的 `fetchBatches()` 接收列表筛选参数，默认：

```ts
{ bean_archive_status: 'active' }
```

切换筛选时重新调用后端，不要在已经分页的 20 条结果上做客户端归档过滤。

### 9.3 页面筛选

在 `RoastingBatchList.vue` 增加：

```vue
<select v-model="filters.beanArchiveStatus" @change="fetchBatches">
  <option value="active">在用生豆批次</option>
  <option value="archived">已归档生豆批次</option>
  <option value="all">全部历史批次</option>
</select>
```

默认值：

```ts
beanArchiveStatus: 'active'
```

归档行显示标签：

```vue
<span v-if="b.greenBeanIsArchived" class="batch-source-tag">已归档生豆</span>
```

### 9.4 数量显示

页脚应显示后端 `total`，而不是只显示当前客户端 `filteredBatches.length`：

```text
共 3 条，当前页 3 条
```

当有分页时，客户端长度不是总数。

## 10. 修改五：明确不同页面的数据范围（P1）

逐个检查调用 `/roasting-batches` 的页面：

| 页面/用途 | 推荐范围 |
| --- | --- |
| 烘焙分析默认列表 | `active` |
| 新建待烘计划下拉框 | `active` |
| 归档历史筛选 | `archived` |
| 全部历史筛选 | `all` |
| 某个批次详情直达 | 不受列表过滤影响 |
| 评价/复盘历史详情 | 允许归档数据 |
| 当前运营 Dashboard | 产品确认后建议 `active` |
| 历史统计报表 | 显式 `all` |

不要通过修改全局 context 为只加载 active 来解决问题，否则归档曲线详情会失去生豆名称。正确做法是：上下文可加载 all，业务列表必须显式声明自己的数据范围。

## 11. 不建议直接删除当前 6 条历史批次

这 6 条记录来自此前联调数据，但它们的外键完整，曲线也能正常查看。直接 SQL 删除可能破坏：

- 库存流水；
- 原始曲线文件；
- 当前生效曲线；
- 问卷和评价；
- 复盘；
- 提醒；
- 批量导入审计记录。

推荐先完成归档过滤，使默认页面只显示 3 条。

如果后续决定清理测试数据，应单独实现管理员 purge 或开发环境重置流程，先列出影响范围并备份数据库。本指南不执行物理删除。

## 14. 当前数据库校验 SQL

### 14.1 按归档状态统计

```sql
SELECT
  gb.is_archived,
  count(DISTINCT rb.id) AS roast_count,
  count(DISTINCT rc.id) AS active_curve_count,
  count(DISTINCT cf.id) AS curve_file_count
FROM roasting_batches rb
JOIN purchase_batches pb ON pb.id = rb.purchase_batch_id
JOIN green_beans gb ON gb.id = pb.green_bean_id
LEFT JOIN roasting_curves rc ON rc.roasting_batch_id = rb.id
LEFT JOIN curve_files cf ON cf.roasting_batch_id = rb.id
GROUP BY gb.is_archived
ORDER BY gb.is_archived;
```

当前预期：

```text
false: roast=3, active_curve=3, curve_file=3
true:  roast=6, active_curve=6, curve_file=6
```

### 14.2 检查重复生效曲线

```sql
SELECT roasting_batch_id, count(*)
FROM roasting_curves
GROUP BY roasting_batch_id
HAVING count(*) > 1;
```

预期：0 行。

### 14.3 检查孤儿记录

```sql
SELECT rb.id
FROM roasting_batches rb
LEFT JOIN purchase_batches pb ON pb.id = rb.purchase_batch_id
WHERE pb.id IS NULL;

SELECT rc.id
FROM roasting_curves rc
LEFT JOIN roasting_batches rb ON rb.id = rc.roasting_batch_id
WHERE rb.id IS NULL;
```

预期：均为 0 行。
