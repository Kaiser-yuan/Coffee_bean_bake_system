# 咖啡烘焙系统前端文档具体修改指南

> 编写日期：2026-06-11  
> 适用文档：
>
> - `咖啡烘焙系统_前端设计_Summary.md`
> - `咖啡烘焙系统_数据流与操作流.md`
> - `咖啡烘焙系统_功能说明书.md`

## 1. 修改目标

本次修改的目标是统一三份文档中的对象定义、业务规则和操作闭环，避免后续数据库与接口按照错误规则实现。

修改后应确保：

```text
生豆档案 -> 采购批次 -> 烘焙批次 -> 曲线分析
-> 个人初始复盘 -> 问卷评价 -> 综合复盘
-> 下一锅建议与提醒 -> 新待烘计划
```

同时满足：

- 生豆档案、豆种、采购批次、烘焙批次概念不混用。
- 问卷收集不依赖曲线上传或复盘创建。
- 出豆量等实际数据不能被系统错误推导。
- 库存变化能够被完整追溯并始终准确。
- 三份文档对同一业务规则使用相同口径。

## 2. 必须优先修改的问题

### 2.0 增加豆款匹配确认规则

#### 当前风险

录入生豆时如果仅通过生豆名称判断是否已有档案，可能把不同产季、处理法、品牌或产区的同名豆错误合并。

系统输入名称后只提供已有豆款的匹配建议，不自动判断为同一豆款。候选项至少展示：

```text
名称
品牌
产季
处理法
产区
```

用户需要主动选择：

```text
选择已有豆款 -> 新增采购批次
创建新豆款 -> 创建生豆档案和首个采购批次
```

不同产季原则上视为不同豆款，应分别建立生豆档案。后续跨产季分析通过名称、豆种、产区等字段聚合比较，而不是合并档案。

#### 验收标准

- 系统不会仅根据名称自动合并生豆档案。
- 同名候选项能够清楚展示品牌、产季、处理法和产区。
- 用户可以明确选择已有豆款或创建新豆款。

### 2.1 修正出豆量与失重率规则

#### 当前问题

`数据流与操作流`中描述为：

```text
标记完成
-> beanWeightOut 计算出豆量
-> weightLossRate 计算失重率
```

出豆量是烘焙后的实际称重结果，系统无法自动计算。

#### 正确规则

```text
标记完成
-> 弹出轻量确认
-> 确认实际烘焙时间与实际投豆量
-> 记录批次状态为已完成
-> 记录实际烘焙时间
-> 确认实际投豆量
-> 扣减采购批次库存

后续补录出豆量
-> 当投豆量和出豆量均存在时
-> 系统自动计算失重率
```

计算公式：

```text
失重率 = (投豆量 - 出豆量) / 投豆量 * 100%
```

#### 需要修改的文档

- `咖啡烘焙系统_数据流与操作流.md`
  - 修改“烘焙执行流”中的批次状态变更描述。
  - 修改“涉及的数据对象”中标记完成的字段变更。
- `咖啡烘焙系统_功能说明书.md`
  - 明确出豆量允许后续补录。
  - 明确失重率仅在投豆量和出豆量都存在时显示。

#### 验收标准

- 文档中不再出现“自动计算出豆量”。
- 标记完成不要求立即填写出豆量、CSV、评价或复盘。
- 从系统总览勾选完成时，会先确认实际烘焙时间和实际投豆量。
- 出豆量为空时，失重率显示为未录入或待计算。

### 2.2 修正问卷发起入口和前置条件

#### 当前问题

当前流程要求：

```text
批次复盘页
-> 条件：已有曲线数据
-> 发起公开评价问卷
```

这会导致未上传 CSV 或尚未创建复盘的已完成批次无法收集评价。

#### 正确规则

问卷必须关联具体烘焙批次，但不依赖曲线或复盘：

```text
烘焙批次已完成
-> 可以直接发起评价问卷
```

允许的入口：

```text
烘焙批次详情 / 曲线分析页 -> 发起评价问卷
批次复盘页 -> 发起评价问卷快捷入口
```

禁止的入口：

```text
评价管理 -> 创建问卷
```

评价管理只负责汇总、跟踪、关闭和分析。

#### 需要修改的文档

- `咖啡烘焙系统_数据流与操作流.md`
  - 删除“已有曲线数据”前置条件。
  - 将问卷创建动作改为从烘焙批次发起。
  - 保留复盘页作为快捷入口，但不能作为唯一入口。
- `咖啡烘焙系统_功能说明书.md`
  - 在烘焙批次或曲线分析页面增加“发起评价”操作。
  - 保留评价反馈摘要区域中的问卷快捷入口。
- `咖啡烘焙系统_前端设计_Summary.md`
  - 继续保持“问卷必须从具体烘焙批次发起”的总规则。

#### 验收标准

- 已完成批次即使没有 CSV 和复盘，也能发起问卷。
- 问卷始终自动关联当前烘焙批次。
- 评价管理中不存在创建问卷入口。

### 2.4 补完整库存重新计算规则

#### 当前问题

当前文档只明确标记完成时扣减库存，未覆盖逆向操作和修改操作。

#### 正确库存公式

```text
采购批次剩余库存
= 采购总量
- 该采购批次下有效且已完成烘焙批次的投豆量合计
+ 库存调整量合计
```

库存不是手工维护字段，也不依赖增量加减结果，应随事实数据实时重新计算。

#### 必须覆盖的触发场景

| 操作 | 库存结果 |
|---|---|
| 新增采购批次 | 增加可用库存 |
| 创建待烘计划 | 不扣库存 |
| 标记烘焙完成 | 扣减实际投豆量 |
| 取消完成状态 | 恢复投豆量 |
| 作废或删除已完成批次 | 恢复投豆量 |
| 修改已完成批次投豆量 | 重新计算库存 |
| 更换已完成批次采购来源 | 原采购批次恢复，新采购批次扣减 |
| 新增库存调整记录 | 根据调整量重新计算 |

#### 需要修改的文档

- `咖啡烘焙系统_数据流与操作流.md`
  - 扩展库存变化触发时机表。
  - 明确库存始终由后端聚合计算。
- `咖啡烘焙系统_功能说明书.md`
  - 明确剩余库存只读。
  - 明确库存调整记录用于盘点、损耗和取样等修正。

#### 验收标准

- 文档覆盖所有正向和逆向库存变化。
- `remainingStock` 不作为采购批次的人工可编辑事实字段。
- Mock 数据可以暂存 `remainingStock` 用于展示，但需标注其为派生值。

## 3. 对象关系与概念口径修正

### 3.1 修正关系基数

#### 当前问题

当前关系写成：

```text
烘焙批次 1 -> 1 烘焙曲线
烘焙批次 1 -> 1 批次复盘
批次复盘 1 -> 3 烘焙提醒
```

批次创建后，曲线、复盘和提醒都可能尚不存在。

#### 推荐关系

```text
生豆档案 1 -> N 采购批次
采购批次 1 -> N 烘焙批次

烘焙批次 1 -> 0..N 原始曲线文件
烘焙批次 1 -> 0..1 当前生效曲线

烘焙批次 1 -> 0..N 评价问卷
评价问卷 1 -> 0..N 杯测评价记录

烘焙批次 1 -> 0..1 批次复盘
批次复盘 1 -> 0..3 烘焙提醒

采购批次 1 -> 0..N 库存调整记录
```

#### 原始曲线文件与当前生效曲线

同一烘焙批次允许重复上传 CSV：

```text
历史原始 CSV 文件永久保留
-> 选择一份解析结果作为当前生效曲线
-> 后续解析规则更新时可重新解析
```

#### 需要修改的文档

- 三份文档中的对象关系说明。

### 3.2 统一“豆款”和“豆种”的含义

#### 统一定义

```text
生豆档案 / 豆款：
一款具体可采购和烘焙的咖啡豆，例如“耶加雪菲 G1 水洗 2025 产季”。

豆种：
咖啡植物品种，例如 Gesha、Bourbon、Typica、埃塞原生种。
```

#### 必须修改的表述

- 待烘豆列表应按`生豆档案 / 豆款`分组，不应按豆种分组。
- 年度`已烘豆款数`应统计生豆档案 ID 去重数。
- 年度`豆种分布`才按 variety 字段统计。
- 变量 `totalBeanVarieties` 实际表达豆款数，建议改为：

```text
totalRoastedBeanProfiles
```

或：

```text
totalRoastedBeanProducts
```

#### 需要修改的文档

- `咖啡烘焙系统_数据流与操作流.md`
- `咖啡烘焙系统_功能说明书.md`
- 所有 Mock 字段和前端文案

### 3.3 去除烘焙批次中的重复生豆关联

#### 当前问题

Mock `RoastingBatch` 同时保存：

```text
purchaseBatchId
greenBeanId
```

采购批次已经唯一关联生豆档案，重复保存会产生冲突风险。

#### 推荐规则

数据库事实关系：

```text
RoastingBatch.purchaseBatchId
-> PurchaseBatch.greenBeanId
-> GreenBean
```

烘焙批次只需要保存 `purchaseBatchId`。前端需要展示生豆信息时，由后端聚合响应返回。

#### 示例响应

```json
{
  "id": "rb_1",
  "purchaseBatchId": "pb_1",
  "greenBean": {
    "id": "gb_1",
    "name": "耶加雪菲 G1",
    "variety": "埃塞原生种"
  }
}
```

#### 需要修改的文档

- `咖啡烘焙系统_数据流与操作流.md` Mock 数据结构。
- 后续数据库设计与接口契约。

## 4. 评价闭环补充

### 4.1 增加冲煮方式

#### 原因

手冲、杯测、意式等冲煮方式会显著影响风味评分。若不记录，跨评价比较容易产生误判。

#### 问卷增加字段

```text
冲煮方式：
- 杯测
- 手冲
- 意式
- 奶咖
- 其他

饮用温度：
- 热饮
- 冷饮

饮用形式：
- 黑咖啡
- 加奶
- 其他
```

第一版可从标准词表选择，允许选择“其他”并填写备注。

#### 评价记录应保存

```text
评价时间
冲煮方式
评价者类型
各维度评分
风味标签
自由备注
```

系统根据评价时间和烘焙时间自动计算养豆天数。

#### 需要修改的文档

- `咖啡烘焙系统_功能说明书.md`
  - 公开问卷字段。
  - 评价明细字段。
- `咖啡烘焙系统_数据流与操作流.md`
  - 评价提交和统计流程。

---

### 4.2 允许问卷维度选择“未评价”

#### 当前风险

评价者可能无法判断所有维度。例如拿到已经冲煮好的咖啡时，无法评价干香；强制填写会产生虚假数据。

#### 推荐规则

每个评分维度支持：

```text
1、2、3、4、5
未评价 / 不确定
```

统计规则：

```text
平均分、分布和样本量统计时
-> 忽略“未评价 / 不确定”
-> 每个维度单独展示有效评价数量
```

综合喜好可以保持必填，其他风味维度允许未评价。

#### 验收标准

- 评价者可以跳过无法判断的维度。
- 页面明确展示每个维度的有效评价数。
- 未评价数据不按 0 分参与平均值计算。

---

### 4.3 统一评分精度

#### 当前问题

文档描述“支持半星视觉”，但实际数据为 `1-5` 整数分，容易让用户误以为能够提交半星。

#### 推荐规则

第一版统一使用整数五分制：

```text
可选值：1、2、3、4、5
不支持半星提交
```

若视觉上显示半星，应仅用于展示平均分，不应用于填写。

#### 需要修改的文档

- `咖啡烘焙系统_功能说明书.md`
  - 将公开问卷描述改为“整数五星评分”。
  - 评价结果平均分允许显示小数和半星视觉。

## 6. 补充实施规则

### 6.1 批次状态与资料完整度分开

批次状态和资料完整度必须分开，避免将“实际烘焙是否发生”和“资料是否齐全”混为一谈。

```text
批次状态：
- 待烘焙
- 已完成
- 已作废

资料完整度：
- 缺出豆量
- 缺 CSV
- 缺评价
- 缺复盘
- 资料完整
```

批次完成只表示实际烘焙已经发生，不代表其他资料全部齐全。系统总览可根据资料完整度生成待处理事项。

### 6.2 第一版曲线文件管理

长期可以支持一个烘焙批次关联多份原始 CSV，但第一版不实现复杂版本管理。

```text
一个烘焙批次
-> 一份当前使用的 CSV
-> 一份当前生效的解析结果

重新上传
-> 替换当前解析结果
-> 原始上传文件保留备份
```

拿到真实 M1 CSV 并出现实际版本管理需求后，再扩展多文件和生效版本模型。

### 6.3 库存流水解释

库存由后端实时聚合计算，但必须提供可解释的库存流水：

```text
采购入库：+5000g
烘焙批次 A：-500g
盘点调整：-20g
当前库存：4480g
```

库存详情需要明确每一笔变化的来源、时间和关联对象。

### 6.4 采购价格定义

采购批次需要明确采购总价与单价（元/kg）。第一版至少录入其中一个，系统根据采购重量自动计算另一个，货币单位默认人民币。

### 6.5 第一版复盘管理

第一版保持一个烘焙批次最多一份批次复盘，内部记录个人初始复盘、综合复盘、下一锅建议、最多三条提醒、创建时间和最后更新时间。

第一版不实现复杂复盘版本历史。出现多人协作或明确审计需求后，再增加版本表。

## 7. 修正后的完整闭环

### 7.1 生豆数据采集闭环

```text
搜索生豆档案
-> 系统展示同名候选豆款
-> 用户根据品牌、产季、处理法和产区确认
-> 已存在：新增采购批次
-> 不存在：创建生豆档案并新增首个采购批次
-> 形成采购库存
-> 通过三级表格追踪采购和烘焙消耗
-> 通过库存调整记录修正实际差异
```

### 7.2 烘焙曲线分析闭环

```text
选择采购批次
-> 创建待烘计划
-> 完成实际烘焙
-> 确认实际时间与投豆量
-> 标记完成并扣减库存
-> 后续补充出豆量并计算失重率
-> 上传和解析 M1 CSV
-> 查看单锅曲线或多锅对比
-> 填写个人初始复盘
```

### 7.3 评价与迭代闭环

```text
从已完成烘焙批次发起问卷
-> 收集不同养豆阶段、冲煮方式和饮用场景的评价
-> 汇总评分与风味反馈
-> 完成综合复盘
-> 形成下一锅建议和最多三条提醒
-> 创建下一次待烘计划
-> 开始新的迭代
```

### 7.4 非阻塞原则

以下操作均允许后续补录：

- 出豆量
- M1 CSV
- 个人初始复盘
- 评价问卷
- 综合复盘

除批次标记完成外，其他步骤不应互相强制阻塞。

## 8. 建议修改顺序

### 第一轮：修正业务事实

1. 增加豆款匹配确认规则。
2. 修正出豆量与失重率规则。
3. 修正问卷发起条件和入口。
4. 补齐生豆与采购字段。
5. 补齐库存重新计算规则。

### 第二轮：统一对象和字段

1. 修正对象关系基数。
2. 统一豆款与豆种术语。

4. 补充评价记录的冲煮方式。
5. 补充饮用场景。
6. 允许评分维度未评价。
7. 统一评分精度。

### 第三轮：统一三份文档

1. 更新前端设计 Summary。
2. 更新数据流与操作流。
3. 更新功能说明书。
4. 对照检查所有页面、Mock 数据和未来接口契约。

## 9. 最终验收清单

### 对象关系

- [ ] 生豆档案与豆种不混用。
- [ ] 生豆档案不会仅根据名称自动合并。
- [ ] 不同产季原则上建立不同豆款档案。
- [ ] 生豆档案 `1 -> N` 采购批次。
- [ ] 采购批次 `1 -> N` 烘焙批次。
- [ ] 烘焙批次只通过采购批次关联生豆。
- [ ] 曲线、复盘、问卷和提醒使用可选基数。

### 生豆与库存

- [ ] 生豆档案包含品牌、产季、豆商风味描述。
- [ ] 采购批次包含采购价格。
- [ ] 采购价格明确区分总价和单价，并可互相计算。
- [ ] 库存不可直接编辑。
- [ ] 取消完成、作废、修改投豆量时库存能够恢复或重算。
- [ ] 库存详情可以解释每一笔库存变化。

### 烘焙批次

- [ ] 标记完成不会自动生成出豆量。
- [ ] 标记完成前确认实际烘焙时间和实际投豆量。
- [ ] 出豆量允许后续补录。
- [ ] 失重率只在数据完整时计算。
- [ ] CSV、评价和复盘不会阻塞批次完成。
- [ ] 批次状态与资料完整度分开显示。

### 评价与复盘

- [ ] 已完成批次无需 CSV 或复盘即可发起问卷。
- [ ] 评价管理不能创建问卷。
- [ ] 问卷记录冲煮方式。
- [ ] 问卷记录必要的饮用场景。
- [ ] 养豆天数由系统自动计算。
- [ ] 无法判断的评分维度允许选择未评价。
- [ ] 未评价不按 0 分参与统计。
- [ ] 填写评分只允许整数，平均分允许小数展示。
- [ ] 个人初始复盘不会被综合复盘覆盖。
- [ ] 下一次提醒最多三条。
- [ ] 第一版每个批次最多一份复盘，不实现复杂版本管理。

### 文档一致性

- [ ] 三份文档使用相同对象名称。
- [ ] 三份文档使用相同关系基数。
- [ ] 三份文档使用相同库存和评价规则。
- [ ] Mock 数据结构与未来接口契约一致。
