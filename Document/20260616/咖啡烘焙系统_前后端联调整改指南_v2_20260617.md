# 咖啡烘焙系统前后端联调整改指南 v2

日期：2026-06-17

## 1. 当前结论

生豆录入链路已经不是“网络未连接”问题。当前真实情况是：

- 后端服务可以启动，登录接口可用。
- `POST /api/v1/green-beans/with-first-purchase` 可以成功创建生豆和首批采购。
- `GET /api/v1/green-beans/tree` 在不带筛选参数时可以返回列表。
- `GET /api/v1/green-beans/tree?variety=Typica` 或前端顶部品种/处理法筛选不为空时会触发 500。
- 前端 `npm run build` 已通过，说明当前主要阻塞在后端筛选查询逻辑，而不是前端编译问题。

因此，如果你录入时页面顶部筛选条件不为空，可能出现一种很迷惑的现象：

1. 后端其实已经创建成功。
2. 前端保存后调用 `fetchData()` 刷新列表。
3. 刷新列表因为筛选参数触发后端 500。
4. 前端 catch 到异常后显示“保存失败”或进入错误状态。

这会让人误以为“录入失败”，但真正失败的是保存后的列表刷新。

## 2. 必须修改 1：后端生豆树筛选不能用文本值比较 UUID 字段

文件：

`backend/app/repositories/green_beans.py`

问题位置：

```python
if variety:
    stmt = stmt.where(GreenBean.variety_term_id == variety)
if process:
    stmt = stmt.where(GreenBean.process_term_id == process)
```

问题原因：

- `GreenBean.variety_term_id` 和 `GreenBean.process_term_id` 是 UUID 字段。
- 前端筛选传过来的是文本值，例如 `Typica`、`Washed`、`铁皮卡`、`水洗`。
- PostgreSQL 会尝试把 `Typica` 当 UUID 解析，结果报错：

```text
invalid UUID 'Typica': length must be between 32..36 characters
```

建议改法：

先在 import 中加入 `StandardTerm`：

```python
from ..models.all_models import GreenBean, PurchaseBatch, RoastingBatch, StandardTerm
```

然后把筛选逻辑改成按关联标准术语的 `value` 查询：

```python
if variety:
    stmt = stmt.where(GreenBean.variety.has(StandardTerm.value == variety))
if process:
    stmt = stmt.where(GreenBean.process.has(StandardTerm.value == process))
```

修改后这一段应类似：

```python
if search:
    stmt = stmt.where(
        GreenBean.name.ilike(f"%{search}%") |
        GreenBean.brand.ilike(f"%{search}%") |
        GreenBean.region.ilike(f"%{search}%")
    )
if variety:
    stmt = stmt.where(GreenBean.variety.has(StandardTerm.value == variety))
if process:
    stmt = stmt.where(GreenBean.process.has(StandardTerm.value == process))
if region:
    stmt = stmt.where(GreenBean.region == region)
```

说明：

- 这个改法匹配当前前端行为，因为前端传的是文本值。
- 另一种方案是前端传标准术语 id，但这会牵涉筛选下拉数据结构，当前阶段不建议扩大修改面。

## 3. 必须修改 2：修改后要硬重启后端，不要只依赖 reload

本次排查中，即使日志出现：

```text
WatchFiles detected changes in 'app\\repositories\\green_beans.py'. Reloading...
```

后续请求里仍然出现旧 SQL：

```sql
WHERE green_beans.variety_term_id = $1::UUID
```

这说明当前开发进程可能没有真正切到最新代码，或者旧 worker 仍在处理请求。

建议你修改完后，在 PowerShell 中执行：

```powershell
Get-Process -Name uvicorn,uv -ErrorAction SilentlyContinue | Stop-Process -Force
```

如果 8000 端口仍然被占用，再查端口进程：

```powershell
Get-NetTCPConnection -LocalPort 8000 -ErrorAction SilentlyContinue | Select-Object LocalAddress,LocalPort,State,OwningProcess
```

然后按实际 `OwningProcess` 停掉：

```powershell
Stop-Process -Id <OwningProcess> -Force
```

重新启动后端：

```powershell
cd E:\Code_development\Coffee_bake_system\backend
& 'C:\Users\Kaiser\AppData\Local\Programs\Python\Python312\Scripts\uv.exe' run uvicorn app.main:app --host 127.0.0.1 --port 8000 --reload
```

## 5. 建议修改：前端保存成功和刷新失败要分开提示

文件：

`frontend/src/views/GreenBeanManagement.vue`

当前逻辑：

```ts
try {
  // 保存
  await greenBeanApi.createGreenBeanWithFirstPurchase(...)
  await fetchData()
  closeDrawer()
} catch (err) {
  alert(err instanceof ApiError ? err.message : '保存失败')
}
```

风险：

- 如果保存成功，但 `fetchData()` 失败，前端仍会提示“保存失败”。
- 这会掩盖真实状态，导致重复录入测试数据。

建议改成两段：

```ts
async function onSubmit() {
  try {
    if (isDemoMode) {
      // 保留现有 demo 分支
    } else {
      if (editMode.value) {
        await greenBeanApi.addPurchaseBatch(
          selectedBeanId.value,
          toPurchaseBatchCreateDto(form.value),
        )
      } else {
        await greenBeanApi.createGreenBeanWithFirstPurchase(
          toGreenBeanWithFirstPurchaseDto(form.value),
        )
      }
    }
  } catch (err) {
    alert(err instanceof ApiError ? err.message : '保存失败')
    return
  }

  closeDrawer()

  try {
    await fetchData()
  } catch {
    alert('保存已成功，但刷新列表失败。请手动刷新页面或清空筛选后重试。')
  }
}
```

注意：

- 这不是后端 500 的根因修复。
- 真正必须先修的是第 2 节的后端筛选逻辑。
- 这个前端改法只是避免“保存成功但刷新失败”时误导用户。

## 6. 建议检查：筛选值和标准术语值是否一致

文件：

`frontend/src/types.ts`

需要确认：

- `VARIETY_OPTIONS` 里的值，要和后端 `standard_terms` 表中 `category='variety'` 的 `value` 一致。
- `BEAN_PROCESSES` 里的值，要和后端 `standard_terms` 表中 `category='process'` 的 `value` 一致。

如果前端传 `铁皮卡`，后端标准术语中也应该是 `铁皮卡`。

如果前端传 `Typica`，后端标准术语中也应该是 `Typica`。

否则筛选不会报错，但会查不到数据。
