## 3. P0：修复真实 Kaleido CSV 解析器

### 3.1 当前问题

当前 `backend/app/parsers/kaleido_m1.py` 按另一种 CSV 格式实现：

- 参数按 `key=value` 读取。
- 事件按逗号分隔读取。
- 事件名按 `CHARGE`、`FCs` 等读取。
- `Index` 被读取为小写 `index`。
- BT、ET、RoR 和 SV 被额外除以 `10`。

真实文件格式为：

```text
KLDO data file V101

[{PARAMETERS}]
[{TotalTime}]
06:25

[{PreTemp}]
206

[{EVENT}]
[{StartBeansIn}]
206@00:00

[{TemperBack}]
111@00:59

[{DATA}]
Index,Time,BT,ET,RoR,SV,HPM,HP,SM,RL,PS
```

当前实现会造成：

- 参数为空。
- 事件为空。
- 采样序号全部为 `0`。
- 温度和 RoR 只有真实值的十分之一。

### 3.2 修改方案

建议重构解析器为四个明确步骤：

```text
identify_format
-> split_sections
-> parse_marker_value_pairs
-> parse_data_rows
```

推荐函数：

```python
def split_kldo_sections(lines: list[str]) -> dict[str, list[str]]:
    ...

def parse_marker_value_section(lines: list[str]) -> dict[str, str]:
    ...

def parse_event_value(value: str) -> tuple[float, float] | None:
    ...

def parse_data_rows(lines: list[str]) -> list[KaleidoDataPoint]:
    ...
```

参数映射：

```text
TotalTime -> total_time
PreTemp -> preheat_temp
CookDate -> cooked_at
Comment -> comment
```

事件映射：

```text
StartBeansIn  -> charge
TemperBack    -> turning_point
TurntoYellow  -> yellowing
1stBoomStart  -> first_crack_start
1stBoomEnd    -> first_crack_end
2ndBoomStart  -> second_crack_start
2ndBoomEnd    -> second_crack_end
BeansColdDown -> drop
```

DATA 规则：

```text
sample_index = int(row["Index"])
elapsed_seconds = float(row["Time"]) / 1000
BT = float(row["BT"])
ET = float(row["ET"])
RoR = float(row["RoR"])
SV = float(row["SV"])
```

不得再除以 `10`。

RoR 为 `0` 时：

- 原始值可以保留。
- 前端绘图响应中可以转换为 `null`，避免前段绘制零值曲线。

## 5. P0：修复异步 ORM 关系加载

### 5.1 当前问题

`_to_batch_response()` 和 `compute_batch_completeness()` 会读取：

- `purchase_batch.green_bean`
- `roast_level`
- `active_curve`
- `curve_files`
- `questionnaires`
- `review`

但部分查询使用基础 `get_by_id()`，没有预加载这些关系。AsyncSession 中访问未加载关系可能触发：

```text
sqlalchemy.exc.MissingGreenlet
```

### 5.2 修改方案

不要在 Router 中使用通用 `get_by_id()` 后直接构建复杂响应。

统一使用：

```python
await roasting_batch_repository.get_detail(batch_id)
```

`get_detail()` 和 `get_list()` 应预加载：

```python
selectinload(RoastingBatch.purchase_batch).selectinload(PurchaseBatch.green_bean)
selectinload(RoastingBatch.roast_level)
selectinload(RoastingBatch.active_curve)
selectinload(RoastingBatch.curve_files)
selectinload(RoastingBatch.questionnaires)
selectinload(RoastingBatch.review)
selectinload(RoastingBatch.reminders)
```

建议将响应转换集中到 Service：

```text
RoastingBatchPresenter / build_roasting_batch_response
```

避免多个 Router 重复依赖 ORM 关系状态。

### 5.3 验收

- 获取批次列表不会产生 `MissingGreenlet`。
- 完成、重开、作废后能够直接返回完整批次响应。
- 资料完整度与详情数据一致。

## 6. P0：修复库存计算与流水

### 6.1 当前问题一：采购入库流水余额双倍

当前剩余库存计算已经包含：

```text
PurchaseBatch.total_weight_grams
```

创建采购批次后再次通过 `purchase_in` 增加采购重量，导致库存流水中的 `resulting_grams` 显示双倍。

### 6.2 当前问题二：负数库存调整被写成增加

当前调用：

```python
required_grams=abs(body.amount_grams)
event_type="adjustment"
```

库存服务对非 `roast_consumption` 统一做加法。因此负数调整会：

- 库存聚合正确减少。
- 库存流水错误显示增加。

库存无法通过流水解释，违反核心规则。

### 6.3 推荐方案

将“库存事实计算”和“流水记录”分开：

```python
async def calculate_remaining_stock(...):
    ...

async def append_inventory_ledger(
    db,
    purchase_batch_id,
    change_grams,
    event_type,
    related_entity_type,
    related_entity_id,
):
    ...
```

流水新余额：

```text
resulting_grams = 当前事实库存
```

完成烘焙时：

```text
change_grams = -actual_input_weight_grams
```

重开或作废已完成批次时：

```text
change_grams = +actual_input_weight_grams
```

库存调整时：

```text
change_grams = body.amount_grams
```

采购创建时：

```text
change_grams = total_weight_grams
resulting_grams = total_weight_grams
```

不要使用 `abs()`。

### 6.4 并发控制

完成、重开、作废和库存调整必须锁定采购批次：

```python
select(PurchaseBatch)
.where(PurchaseBatch.id == purchase_batch_id)
.with_for_update()
```

事务内完成：

```text
锁定采购批次
-> 重新计算库存
-> 校验
-> 修改批次状态
-> 写入流水
-> 提交
```

### 6.5 幂等

对以下接口使用 `Idempotency-Key` 或数据库唯一约束：

- 完成烘焙批次。
- 重开批次。
- 作废批次。
- 创建库存调整。

至少增加流水唯一约束：

```text
event_type + related_entity_type + related_entity_id
```

避免同一业务事件重复写入。

### 6.6 测试

- 首购 1000g 后库存为 1000g，流水余额为 1000g。
- 完成 200g 后库存和流水余额均为 800g。
- 重开后恢复为 1000g。
- 完成后作废恢复为 1000g。
- 调整 `-100g` 后库存和流水余额均为 900g。
- 调整 `+100g` 后库存和流水余额均为 1100g。
- 两个并发完成请求不能使库存为负数。
- 重复完成请求不能重复扣减。

## 7. P0：修复曲线替换与重新解析

### 7.1 当前问题

替换当前曲线时：

```python
db.delete(active_curve)
```

缺少 `await`。

重新解析接口只重新调用了解析器并修改 `CurveFile.parse_status`，没有：

- 重建派生指标。
- 重建事件和采样点。
- 原子替换当前有效曲线。
- 更新烘焙批次曲线摘要。

### 7.2 修改方案

抽取统一服务：

```python
async def parse_and_activate_curve(
    db: AsyncSession,
    curve_file: CurveFile,
    content: bytes,
) -> RoastingCurve:
    ...
```

上传和重新解析都调用该服务：

```text
解析原始文件
-> 计算事件、阶段、AUC、控制变化和 warnings
-> 创建新的 RoastingCurve
-> 成功后替换旧曲线
-> 更新 RoastingBatch 摘要
-> 标记 CurveFile parsed
```

必须保证：

- 新解析失败时，旧有效曲线不被删除。
- 新解析成功后才替换旧曲线。
- 原始文件和历史 CurveFile 均保留。

删除 ORM 对象：

```python
await db.delete(active_curve)
```

更推荐使用更新现有 `RoastingCurve` 的方式，避免唯一约束切换问题。

### 7.3 补齐接口

当前设计中还需要：

```text
GET /curve-files/{id}
GET /curve-files/{id}/download
```

下载时：

- 校验文件存在。
- 使用固定 MIME。
- 不信任原始文件名作为磁盘路径。

方案：

1. `users` 增加：

```text
is_admin
```


```python
AdminUserDep
```

## 9. P1：补齐请求校验

当前大量 Schema 仅使用 `int`、`str`，没有范围约束；日期在 Router 中使用 `datetime.fromisoformat()`，错误输入可能变成 500。

推荐直接使用类型和 Field：

```python
from datetime import datetime
from pydantic import Field

class BatchCompleteRequest(BaseModel):
    roasted_at: datetime
    actual_input_weight_grams: int = Field(gt=0)

class EvaluationSubmitRequest(BaseModel):
    overall_preference_score: int = Field(ge=1, le=5)
    dry_fragrance_score: int | None = Field(default=None, ge=1, le=5)
```

需要校验：

- 采购重量和烘焙重量大于零。
- 出豆量大于零且不超过投豆量。
- 含水率范围合理。
- 价格不能为负数。
- 所有评价分数为 `1-5` 或 `NULL`。
- 提醒内容不能为空。
- 最多三条提醒，优先级不重复。
- `review_reminder_ids` 最多三个且必须属于来源复盘。

评价提交需要写入：

```python
brew_method_term_id=body.brew_method
```

并校验词条属于正确分类。

## 12. P1：修复提醒快照关系

当前快照保存 `source_review_reminder_id` 外键，但替换复盘提醒时会删除原提醒。

由于快照应该不可变，应选择：

### 推荐方案

- 保留来源 ID 作为普通 UUID 字段，不设置强外键。
- 快照保存：

```text
source_review_id
source_roasting_batch_id
source_review_reminder_id
priority
content
```

删除或修改来源提醒不能影响已创建批次的提醒快照。

### 同时增加

- 创建下一锅前要求来源批次存在。
- 建议要求来源批次为 `completed`。
- `planned_input_weight_grams > 0`。
- 选择的采购批次默认与来源批次相同；若不同，前端需要明确确认。

## 13. P1：修复 ORM 级联与数据库约束

建议明确外键删除策略和关系级联：

- 生豆存在采购批次时不能删除。
- 采购批次存在烘焙批次时不能删除。
- 问卷删除策略需要保留评价历史。
- 曲线文件历史不能因替换有效曲线被删除。
- 替换复盘提醒不能删除批次提醒快照。

增加数据库约束：

- 评价分数 `1-5`。
- `actual_input_weight_grams > 0`。
- `output_weight_grams > 0`。
- `output_weight_grams <= actual_input_weight_grams` 可由业务层保证。
- 库存调整金额不能为 `0`。
- 问卷分享码唯一。
