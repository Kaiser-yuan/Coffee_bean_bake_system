## 4. 契约策略：前端 camelCase，后端 snake_case

当前前端类型使用：

```text
purchaseBatchId
beanWeightIn
actualDate
developmentRatio
```

后端返回：

```text
purchase_batch_id
actual_input_weight_grams
roasted_at
development_ratio_percent
```

不能在页面中到处手工转换。

### 推荐方案

建立三层：

```text
后端 API DTO（snake_case）
-> Adapter / Mapper
-> 前端领域模型（camelCase）
```

目录：

```text
frontend/src/api/
├── http.ts
├── types.ts
├── auth.ts
├── terms.ts
├── greenBeans.ts
├── purchaseBatches.ts
├── roastingBatches.ts
├── curves.ts
├── questionnaires.ts
├── evaluations.ts
├── reviews.ts
└── dashboard.ts

frontend/src/adapters/
├── greenBean.ts
├── purchaseBatch.ts
├── roastingBatch.ts
├── curve.ts
├── questionnaire.ts
└── review.ts
```

示例：

```ts
export function toRoastingBatch(dto: RoastingBatchDto): RoastingBatch {
  return {
    id: dto.id,
    purchaseBatchId: dto.purchase_batch_id,
    plannedDate: dto.planned_at,
    actualDate: dto.roasted_at,
    beanWeightIn:
      dto.actual_input_weight_grams ??
      dto.planned_input_weight_grams,
    beanWeightOut: dto.output_weight_grams,
    developmentRatio: dto.development_ratio_percent,
    status: dto.status,
  }
}
```

前端页面只能依赖领域模型或 ViewModel，不直接依赖后端 snake_case DTO。

## 5. 前端联调基础设施

### 5.1 HTTP 客户端

由于当前依赖中没有 Axios，可以先使用原生 `fetch`，避免额外依赖。

`frontend/src/api/http.ts` 负责：

- API Base URL。
- JWT Token。
- JSON 请求和响应。
- 文件上传。
- 401 处理。
- 后端统一错误结构。
- 超时和网络错误。

环境变量：

```text
VITE_API_BASE_URL=http://localhost:8000/api/v1
VITE_DEMO_MODE=false
```

生产环境：

```text
VITE_API_BASE_URL=/api/v1
```

### 5.2 认证 Store

新增：

```text
frontend/src/stores/auth.ts
frontend/src/views/Login.vue
```

Store 至少保存：

- `accessToken`
- `expiresAt`
- `userId`
- `displayName`
- `isAuthenticated`

路由规则：

- `/eval/:shareCode` 公开访问。
- `/login` 公开访问。
- 其他路由需要登录。

收到 401：

```text
清除 Token
-> 跳转登录页
-> 保留 redirect 参数
```

### 5.3 Mock 与真实 API 切换

不要直接删除 Mock。建议使用 Repository 入口：

```ts
const dataSource =
  import.meta.env.VITE_DEMO_MODE === 'true'
    ? mockRepository
    : apiRepository
```

GitHub Pages：

```text
VITE_DEMO_MODE=true
```

本地联调和正式服务器：

```text
VITE_DEMO_MODE=false
```

这样静态演示与正式系统可以长期共存。

## 11. 错误响应约定

后端业务错误：

```json
{
  "code": "INSUFFICIENT_INVENTORY",
  "message": "库存不足",
  "details": {
    "available_grams": 300,
    "required_grams": 500
  },
  "request_id": "..."
}
```

前端显示原则：

- `message` 给用户。
- `details` 用于字段和确认信息。
- `request_id` 用于问题排查，不作为主要 UI。

需要统一处理：

```text
400：请求错误
401：需要登录
403：没有权限
404：对象不存在
409：状态冲突或重复操作
422：字段校验失败
500：服务异常
```

## 13. 完整闭环验收脚本

使用一个全新数据库执行：

1. 创建首个管理员并登录。

5. 完成烘焙，实际投豆 `280g`，库存变为 `720g`。
6. 补录出豆量 `240g`，自动计算失重率。
7. 上传 `260530_9.csv`，验证曲线指标。
8. 创建问卷并通过公开链接提交评价。
9. 查看评价聚合。
10. 保存个人复盘、综合复盘和三条提醒。
11. 创建下一锅计划 `300g`，复制提醒快照。
12. 修改来源提醒，下一锅快照保持不变。
13. 完成下一锅并上传 `260530_10.csv`。
14. 比较两锅曲线，验证事件时间差。
15. 查看系统总览，所有聚合与明细一致。
