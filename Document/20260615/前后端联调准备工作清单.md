## 联调策略

### 推荐：本机先联调，再部署服务器

| 理由 | 说明 |
|------|------|
| 调试效率高 | 本地改代码 → 重启几秒，服务器需要 SSH + 上传 + 重启 |
| Swagger 方便 | `http://localhost:8000/api/v1/docs` 直接交互式测试所有接口 |
| 无需配网络 | 不需要 Nginx、HTTPS、域名、防火墙 |
| 出问题可回滚 | 本地数据库可以随意重置 |

### 联调顺序（逐模块）

```
1. 标准词表（terms）
2. 生豆与采购（green_beans + purchase_batches）
3. 烘焙批次与库存（roasting_batches + inventory）
4. CSV 与曲线（curves）
5. 问卷与评价（questionnaires + evaluations）
6. 复盘与下一锅（reviews + next-roast-plan）
7. 系统总览（dashboard）
```

每个模块完成后：前端停止读取该模块的 Mock 数组 → 改为调用 API → 刷新页面验证数据持久化 → 验证错误状态

## 准备工作清单

### 第 1 步：安装 Docker Desktop

PostgreSQL 通过 Docker Compose 一键启动，不需要手动安装配置数据库。

**下载**: https://www.docker.com/products/docker-desktop/

安装后验证：

```bash
docker --version
docker compose version
```

### 第 2 步：创建 .env 文件

```bash
cd backend
cp .env.example .env
```

打开 `.env` 文件，至少修改以下值：

```
SECRET_KEY=<生成一个随机字符串>
JWT_SECRET_KEY=<生成另一个随机字符串>
```

其他值保持默认即可（本地开发环境）。

### 第 3 步：创建数据库迁移

```bash
cd backend
pip install -e ".[dev]"

# 生成初始迁移
alembic revision --autogenerate -m "initial_schema"

# 执行迁移（创建所有表）
alembic upgrade head
```

### 第 4 步：启动后端服务

**方式 A — Docker Compose（推荐，PostgreSQL + Backend 一起启动）**:

```bash
cd backend
docker compose up -d
```

**方式 B — 仅启动后端（需要本地已有 PostgreSQL）**:

```bash
cd backend
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
```

### 第 5 步：创建管理员用户

启动后端后，访问 `http://localhost:8000/api/v1/docs`，找到 `POST /api/v1/auth/register`：

```json
{
  "username": "admin",
  "password": "your-password",
  "display_name": "烘焙师"
}
```

返回 JWT token，点击右上角 "Authorize" 按钮，填入 token。

### 第 6 步：初始化标准词表

通过 Swagger 的 `POST /api/v1/admin/terms` 逐个创建初始词表，或通过数据库直接导入种子数据（后端目前没有 seed 脚本，这一步可能需要在联调过程中逐步添加）。

需要初始化的词表分类：
- `variety` — 豆种：铁皮卡、波旁、卡杜拉、瑰夏、SL28、SL34 等
- `process` — 处理法：水洗、日晒、蜜处理、厌氧发酵、半水洗、其他
- `region` — 产区：埃塞俄比亚、肯尼亚、哥伦比亚、云南等
- `flavor` — 风味：花香、柑橘、莓果、巧克力等
- `roast_level` — 烘焙度：浅烘、中浅烘、中烘、中深烘、深烘
- `brew_method` — 冲煮方式：杯测、手冲、意式、奶咖、其他
- `drink_temperature` — 饮用温度：热饮、冷饮
- `drink_form` — 饮用形式：黑咖啡、加奶、其他
- `evaluator_type` — 评价者类型：烘焙师、同事、顾客

### 第 7 步：前端 API 层改造

当前前端直接导入 Mock 数组（`mockGreenBeans`, `mockPurchaseBatches` 等）。

联调时需要：

1. 创建 `frontend/src/api/` 目录
2. 每个模块创建独立的 API 模块文件
3. 集中管理 HTTP 客户端（base URL、Token、错误处理）
4. 页面改为通过 API 模块获取数据

推荐目录结构：

```
frontend/src/api/
├── http.ts          # axios/fetch 封装，base URL, token, 错误处理
├── auth.ts          # POST /auth/login
├── terms.ts         # GET /terms, POST/PATCH /admin/terms
├── greenBeans.ts    # GET /green-beans/matches, POST /green-beans/with-first-purchase
├── purchaseBatches.ts
├── roastingBatches.ts
├── curves.ts
├── questionnaires.ts
├── evaluations.ts
├── reviews.ts
└── dashboard.ts
```

改造策略：**逐一模块替换，不要一次性全部替换**

每个模块的替换流程：
1. 确认后端接口通过 Swagger 测试通过
2. 创建前端 API 模块文件
3. 修改页面：将 `import { mockXxx }` 替换为 `import { apiGetXxx }`
4. 验证：加载成功、加载为空、业务错误、网络错误
5. 进入下一个模块

### 第 8 步：输入示例数据

从 Swagger 或 Postman 逐个创建测试数据：

1. 创建生豆档案 + 首购（`POST /green-beans/with-first-purchase`）
2. 创建待烘计划（`POST /roasting-batches`）
3. 标记完成（`POST /roasting-batches/{id}/complete`）
4. 补录出豆量（`PATCH /roasting-batches/{id}/output-weight`）
5. 上传 CSV（`POST /roasting-batches/{id}/curve-files`）
6. 创建问卷（`POST /roasting-batches/{id}/questionnaires`）
7. 提交评价（`POST /public/questionnaires/{code}/evaluations`）
8. 填写复盘（`PATCH /roasting-batches/{id}/review/personal` 等）
9. 创建下一锅（`POST /roasting-batches/{id}/next-roast-plan`）

## 后续部署到服务器的额外步骤

本地联调全部通过后，服务器部署只需：

1. 在服务器上 `git pull`
2. 配置服务器 `.env`（数据库密码、JWT 密钥、CORS 域名）
3. `docker compose up -d`
4. 配置 Nginx 反向代理（`/api/` → `localhost:8000`，其他 → Vue 静态文件）
5. 配置 HTTPS（Let's Encrypt）
6. 初始化服务器数据库（执行 migration + 创建管理员）
