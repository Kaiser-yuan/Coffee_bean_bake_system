## 4. P0-2：修复后端 Dockerfile 构建顺序

### 当前问题

当前 `backend/Dockerfile` 逻辑是：

```dockerfile
COPY pyproject.toml .
RUN pip install --no-cache-dir .
COPY . .
```

这意味着执行 `pip install .` 时，镜像内还没有 `app/` 目录。对于当前项目结构，这很容易导致后端镜像构建失败。

### 修改建议

先在 `backend/pyproject.toml` 中补充包配置：

```toml
[tool.hatch.build.targets.wheel]
packages = ["app"]
```

再调整 `backend/Dockerfile`：

```dockerfile
FROM python:3.12-slim

WORKDIR /app

COPY pyproject.toml .
COPY app ./app
RUN pip install --no-cache-dir .

COPY alembic ./alembic
COPY alembic.ini ./alembic.ini

RUN mkdir -p data/uploads

EXPOSE 8000

CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

注意：如果后续有 `scripts/`、`tests/`、`seed/`，再按需复制，不要一开始把无关文件都放进生产镜像。

## 6. P0-4：修复异步 ORM 懒加载风险

### 当前问题

当前响应转换中有类似写法：

```python
if hasattr(batch, "purchase_batch") and batch.purchase_batch:
```

在 SQLAlchemy async 场景中，访问未提前加载的关系字段可能触发懒加载，从而出现：

```text
MissingGreenlet
```

尤其是新建烘焙批次后直接返回 `_to_batch_response(batch)`，此时 `purchase_batch.green_bean` 不一定已经 eager load。

Dashboard 中也存在类似风险：查询时只 eager load 了一部分关系，但后面检查 `active_curve`、`questionnaires`。

### 修改建议

采用统一规则：

1. Repository 层负责 eager load。
2. Response 转换函数只读取确定已加载的关系。
3. 不使用 `hasattr()` 触发关系探测。
4. 新增、更新后如果要返回完整响应，重新通过 `repo.get_detail(id)` 查询一次。

例如创建烘焙批次后：

```python
db.add(batch)
await db.flush()

batch = await repo.get_detail(batch.id)
return _to_batch_response(batch)
```

Dashboard 查询应显式加入：

```python
selectinload(RoastingBatch.active_curve)
selectinload(RoastingBatch.questionnaires)
selectinload(RoastingBatch.review)
```

### 无 Docker 验收标准

- [ ] 搜索代码中不再依赖 `hasattr(model, "relationship")` 判断 ORM 关系。
- [ ] 新增、更新、完成、重开、作废烘焙批次后，返回前都重新加载详情或只返回基础字段。
- [ ] Dashboard 所需关系全部显式 `selectinload`。

### 联调验收标准

- [ ] 创建烘焙批次接口不报错。
- [ ] 完成烘焙批次接口不报错。
- [ ] Dashboard 页面加载不报 `MissingGreenlet`。
- [ ] 烘焙批次列表能显示豆名、批次状态、曲线状态、评价状态。
